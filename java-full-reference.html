Basics
Variables
String name = "Hello World";

Declares a variable that can be reassigned later, initialized with a value.

final String name = "Hello World";

Declares a variable that cannot be reassigned, also initialized with a value.

Primitive types

Primitive types are the most basic data types, and they aren't objects. They hold simple, fixed-size values like numbers and booleans, without using references.

byte smallNumber = 100;

Saves memory in large arrays where the values are small. Range: -128 to 127.

short mediumNumber = 30000;

Rarely used, but useful for memory efficiency. Range: -32,768 to 32,767.

int age = 25;

The most commonly used integer type. Range: -2,147,483,648 to 2,147,483,647.

long bigNumber = 9223372036854775807L;

Used when int isn't large enough. Needs an L suffix. Range: roughly ±9.2 quintillion.

float price = 10.99f;

A 32-bit IEEE-754 floating-point number, about 6-7 significant decimal digits of precision (the exact precision depends on the number's size). Needs the f suffix on literals.

double pi = 3.141592653589793;

A 64-bit IEEE-754 floating-point number, about 15-16 significant decimal digits of precision (again depending on magnitude). The standard type for floating-point math.

char letter = 'A';
char unicodeChar = '\u00A9'; // '©'

char holds a single UTF-16 code unit. Most common characters fit in one char, though some Unicode characters need two (a surrogate pair).

boolean isJavaFun = true;

Stores only true or false.

Non-primitive types

Non-primitive (reference) types hold references to objects instead of raw values. The objects those references point to get created at runtime, for instance via new, array creation, string literals, anonymous classes, lambdas, or method returns. A reference can be null when it points to nothing.

String name = "Hello"; // String literal
Point p1 = new Point(10, 20); // Explicit object creation

Strings are sequences of characters, immutable in Java. Object creation typically uses new, as with the Point example.

int[] numbers = {1, 2, 3, 4};       // array of integers
String[] names = new String[3]; // default values: null
String[] names2 = {"Alice", "Bob", "Charlie"}; // predefined values

Arrays are fixed-size containers for elements of one type, and can hold either primitive or non-primitive types.

class Car { String model; int year; }

Car myCar = new Car(); // creating an object
myCar.model = "Tesla";  // assigning values
myCar.year = 2024;

A class is a blueprint for objects; an object is an instance of a class. Objects can have attributes (variables) and methods (functions), and are created with new.

int x = 10;
Integer y = Integer.valueOf(x); // wrapping int into an object
int z = y.intValue();        // unwrapping, back to a primitive

Java offers wrapper classes to treat primitives as objects (Integer for int, Double for double, Boolean for boolean, etc). Useful in collections (ArrayList<Integer>) and for utility methods like Integer.parseInt("100").

ArrayList<String> names = new ArrayList<>();
names.add("Alice");
names.add("Bob");
System.out.println(names.get(0)); // Alice

Collections are data structures that store objects. Many, like ArrayList, can grow or shrink dynamically; others may be fixed-size or immutable depending on how they're created. Common types: ArrayList, LinkedList, HashSet, HashMap.

String a = new String("Hello");
String b = new String("Hello");
System.out.println(a == b);        // false, different memory locations
System.out.println(a.equals(b));   // true, same content

Passing objects to a method passes the reference by value: the method gets a copy of the reference, so it can modify the object but not the caller's reference itself. Use .equals() rather than == for comparison, since == checks reference equality, not value equality. Java's garbage collector automatically clears out unreferenced objects.

Strings
String str1 = "Hello, World!";

String literals are automatically pooled: identical literals share the same instance, which saves memory by reusing existing string values.

String str2 = new String("Hello, World!");

This creates a new object in heap memory even if an identical string already exists in the pool. Not recommended unless you specifically need that.

String name = "Java";
name = name + " 17"; // creates a new String object ("Java 17"), the old one is discarded

Strings are immutable, they cannot be changed after creation. Any modification creates a new object rather than altering the original. Use StringBuilder for efficiency when modifying strings frequently.

String text = "Hello";
text.length();  // 5
text.charAt(1); // 'e'
String s = "Java Programming";
s.substring(5);    // "Programming"
s.substring(0, 4); // "Java" (exclusive end index)

Extracts substrings.

String word = "Java";
word.toUpperCase(); // JAVA
word.toLowerCase(); // java

file.startsWith("doc"); // true
file.endsWith(".pdf");  // true

text2.replace("Java", "Python"); // "I like Python"
String csv = "Apple,Banana,Cherry";
String[] fruits = csv.split(",");
fruits[1]; // "Banana"
String[] words = {"Java", "Python", "C++"};
String result = String.join(", ", words); // "Java, Python, C++", only works with String[]

a.equals(b); // true, case-sensitive; use .equalsIgnoreCase() otherwise
String a2 = "Hello";
String b2 = new String("Hello");
a2 == b2; // false, different objects in memory; == isn't reliable for comparing values
// Method 1

// Method 2

// Method 3, faster for many concatenations

Concatenation can be done with +, concat(), or StringBuilder; StringBuilder is faster for multiple concatenations.

String formatted = String.format("Hello %s, your score is %d", "Alice", 90);
// "Hello Alice, your score is 90"

%s string, %d integer, %f float/double, %.2f float with 2 decimal places.

String a3 = "Java", b3 = "Java";
System.out.println(a3 == b3); // true, both refer to the same object in the String Pool

String c3 = new String("Java").intern();
System.out.println(a3 == c3); // true, intern() moves it into the pool

The String Pool avoids duplicate objects, saving memory.

char[] letters = {'J','a','v','a'};
String word2 = new String(letters); // "Java"
char[] back = word2.toCharArray();
str1.isEmpty(); // checks if a string is empty
str1.isBlank(); // checks if it's empty or only whitespace
str1.trim();    // removes whitespace from both sides
String message = """
    Hello,
    This is a multi-line string.
    It preserves line breaks and formatting!
    """;

A text block (Java 15+) is a multi-line string enclosed in triple double quotes. No need to escape newlines or double quotes, indentation is handled smartly, and line breaks are preserved automatically. The opening and closing """ must each be on their own line. Great for HTML, SQL, or JSON literals.

Character.isLetter('A');        // true

Arrays

An array is a fixed-size data structure storing multiple elements of the same type in contiguous memory.

int[] numbers; // declared, no memory allocated yet
int[] numbers2 = new int[5]; // 5 elements, default values: 0
int[] numbers3 = {10, 20, 30, 40, 50}; // declared and initialized in one step
int[] arr = {5, 10, 15};
arr[0];     // 5
arr[1] = 20; // modifies index 1

Zero-based indexing (0 is the first element); an out-of-bounds index throws ArrayIndexOutOfBoundsException.

int[] numbers4 = {10, 20, 30, 40};
numbers4.length; // 4, the number of elements, not the last index
// 2D array
int[][] matrix = { {1,2,3}, {4,5,6} };
matrix[1][2]; // 6
Arrays.sort(numbers);
int[] copy = Arrays.copyOf(original, original.length);
Arrays.fill(arr, 100);
Arrays.binarySearch(sortedArr, 30);
Arrays.toString(numbers); // use Arrays.deepToString() for multi-dimensional arrays
ArrayList<Integer> list = new ArrayList<>();
list.add(10); list.add(20); list.add(30); // [10, 20, 30]

Since arrays have a fixed size, an ArrayList is used for dynamic resizing, more flexible than a plain array.

public static void printArray(int[] arr){
  for (int num : arr) System.out.print(num + " ");
}
public static int[] createArray(){
  return new int[]{10, 20, 30};
}
int[] arr1 = {1,2,3}, arr2 = {1,2,3};
arr1 == arr2;             // false, compares references
Arrays.equals(arr1, arr2); // true, compares elements

Use Arrays.equals() instead of == to compare arrays; for multi-dimensional arrays use Arrays.deepEquals(), since Arrays.equals() only compares the outer array's references.

A collection is a container for objects: a way to group and manage multiple elements as a single unit (a list of names, a set of IDs, a map of user data). The Java Collections Framework (java.util) makes it easy to work with these data structures.

ArrayLists

An ArrayList is a resizable array from java.util. Unlike a normal array, it grows or shrinks dynamically as elements are added or removed. It stores objects only, though autoboxing helps with primitives.

ArrayList<String> names = new ArrayList<>();
ArrayList<String> names2 = new ArrayList<String>(); // explicit, less common
// Java 9+
List<Integer> numbers = new ArrayList<>(List.of(1,2,3,4)); // mutable copy
names.add("Alice");
String first = names.get(0);      // "Alice"
names.set(1, "Charlie");      // replaces element at index 1
names.remove(0);
int size = names.size();
boolean isEmpty = names.isEmpty();
names.contains("Alice");
names.indexOf("Charlie"); // index, or -1
names.clear();
// Traditional for loop
for (int i=0; i<names.size(); i++) System.out.println(names.get(i));
// Enhanced for loop
for (String name : names) System.out.println(name);
String[] array = {"a","b","c"};
ArrayList<String> list = new ArrayList<>(Arrays.asList(array)); // array to ArrayList
String[] newArray = list.toArray(new String[0]);         // ArrayList to array
Collections.sort(numbers);                          // ascending
Collections.sort(numbers, Collections.reverseOrder()); // descending
names.sort((a,b) -> Integer.compare(a.length(), b.length())); // custom comparator, e.g. by length
people.sort((p1,p2) -> Integer.compare(p1.age, p2.age)); // sorting custom objects by a field
LinkedLists

A LinkedList is a doubly-linked list implementation of List and Deque. Unlike ArrayList, where elements sit in one contiguous memory block, a LinkedList stores each element as a node pointing to the next and previous ones.

List<String> names = new LinkedList<>(); // interface type, recommended
names.add("Alice");      // add at end
names.addFirst("Bob");  // add at beginning
names.addLast("Charlie");
String first = names.get(0);
String last = names.getLast();
names.set(1, "Daniel");
names.remove();      // removes first
names.removeFirst(); names.removeLast();
// Enhanced for-loop
for (String name : names) System.out.println(name);
// With an iterator
Iterator<String> it = names.iterator();
while (it.hasNext()) System.out.println(it.next());
HashSets

A HashSet implements Set and stores elements in no particular order, using a hash table. No duplicates are allowed: it internally uses a HashMap, uses each element's hash code to decide where to store it, and silently ignores duplicates.

Set<String> names = new HashSet<>(); // recommended: interface type
names.add("Alice"); names.add("Bob");
names.add("Alice"); // duplicate, won't be added
names.remove("Bob");
names.contains("Alice");
names.size(); names.clear(); names.isEmpty();
List<String> list = Arrays.asList("a","b","a");
Set<String> set = new HashSet<>(list); // removes the duplicate "a"
List<String> newList = new ArrayList<>(set); // back to a list, if you need indexing
TreeSets

A TreeSet is a sorted set that automatically keeps its elements in ascending order by default, and ensures uniqueness. It's implemented with a Red-Black Tree, so add/remove/search all run in O(log n). Nulls aren't allowed.

TreeSet<String> names = new TreeSet<>();
names.add("Alice"); names.add("Charlie"); names.add("Bob");
names.add("Alice"); // duplicate, won't be added
// [Alice, Bob, Charlie]

names.higher("Bob");    // "Charlie", next bigger element
names.lower("Charlie"); // "Bob", next smaller element
names.first();        // "Alice", smallest
names.last();         // "Charlie", largest
TreeSet<String> reverseOrder = new TreeSet<>(Collections.reverseOrder());
reverseOrder.add("Java"); reverseOrder.add("Python"); reverseOrder.add("C++");
// [Python, Java, C++]

TreeSet elements can be sorted in reverse order or with a custom comparator.

LinkedHashSets

A LinkedHashSet only stores unique elements (like a HashSet), but maintains insertion order and offers constant-time performance for add/remove/contains.

LinkedHashSet<String> languages = new LinkedHashSet<>();
languages.add("Java"); languages.add("Python"); languages.add("C++");
languages.add("Java"); // won't be added again
// [Java, Python, C++]
Iterator<String> iterator = languages.iterator();
while (iterator.hasNext()){
  String lang = iterator.next();
}

Using an iterator gives more control, e.g. removing items while iterating.

List<String> reversed = new ArrayList<>(languages);
Collections.reverse(reversed);

LinkedHashSet doesn't support reverse iteration natively, since it isn't index-based like a list. Convert it to a list (as above), or use LinkedList/TreeSet instead.

PriorityQueues

A PriorityQueue orders elements by priority, not just insertion order: natural ordering by default (lowest first), or a custom Comparator. It implements Queue and uses a binary heap internally, think of it like a to-do list that always hands you the most important task first. Not thread-safe (use PriorityBlockingQueue for that); nulls aren't allowed, but duplicates are.

PriorityQueue<Integer> pq = new PriorityQueue<>();
pq.add(10); pq.add(5); pq.add(20); // [5, 10, 20] internally organized
pq.offer(15); // like add(), but returns false instead of throwing on failure
pq.peek(); // 5, highest priority (lowest value), without removing
pq.poll(); // 5, removes and returns it
pq.remove(15); // removes a specific element
pq.contains(10); pq.isEmpty(); pq.size(); pq.clear();
while (!pq.isEmpty()) System.out.println(pq.poll()); // priority order, but empties the queue
PriorityQueue<Integer> copy = new PriorityQueue<>(pq); // clone before polling to keep the original intact
HashMaps

A HashMap<K,V> lets you store key-value pairs. It computes a hash code from the key to decide which bucket to store the value in; if two keys hash to the same bucket, a linked list or tree resolves the collision.

Map<String,Integer> map = Map.of("apple",3, "banana",5, "orange",2); // immutable

Map<String,Integer> map2 = new HashMap<>();
map2.put("apple", 2);
map2.put("apple", 3); // overwrites the previous value
map2.get("apple");          // 3
map2.containsKey("apple");   // true
map2.containsValue(3);      // true
map2.remove("apple");
map2.size(); map2.clear();
for (String key : map.keySet()) { }
for (Integer value : map.values()) { }
for (Map.Entry<String,Integer> entry : map.entrySet()) {
  System.out.println(entry.getKey() + " = " + entry.getValue());
}
TreeMaps

A TreeMap stores key-value pairs like a HashMap, but automatically sorts the keys in natural order (alphabetically or numerically). It's implemented with a Red-Black Tree and implements NavigableMap (which extends SortedMap and Map).

TreeMap<String,Integer> map = new TreeMap<>();
map.put("Banana",2); map.put("Apple",5); map.put("Cherry",3);
map.put("Date",4);
map.put("Apple",10); // updates the value for "Apple"
map.get("Cherry");       // 3
map.remove("Banana");
map.containsKey("Apple"); map.containsValue(4);
map.size(); map.isEmpty(); map.clear();
map.keySet();   // sorted: [Apple, Cherry, Date]
map.values();   // in key order
map.firstKey(); // smallest key: "Apple"
map.lastKey();  // largest key: "Date"
map.higherKey("Cherry"); // next key after: "Date"
map.lowerKey("Cherry");  // previous key before: "Apple"
LinkedHashMaps

A LinkedHashMap stores key-value pairs like a HashMap but maintains insertion order. It allows null keys/values, offers O(1) get()/put(), and has an optional access-order mode, useful for LRU caches.

LinkedHashMap<String,Integer> map = new LinkedHashMap<>();
map.put("Apple",1); map.put("Banana",2); map.put("Cherry",3);
map.put("Date",4);
map.put("Apple",5); // updates the value for "Apple"
map.get("Banana");   // 2
map.remove("Cherry");
map.containsKey("Apple"); map.containsValue(5);
map.size(); map.isEmpty();
// iterates in insertion order
for (String key : map.keySet()) { }
for (Integer value : map.values()) { }
for (var entry : map.entrySet()) { }
LinkedHashMap<String,Integer> accessMap =
  new LinkedHashMap<>(16, 0.75f, true);
accessMap.get("Banana"); // moves "Banana" to the end
  • 16: initial capacity, how many buckets the map starts with internally. Doesn't cap how many entries you can add, it just reduces early resizing.
  • 0.75f: load factor, the map resizes (doubles) once it's 75% full. Lower means more memory but faster access; higher is more compact but can cost performance. 0.75f is the default and recommended in most cases.
  • true: ordering mode, true = access order (a get() or put() moves an entry to the end), false = insertion order. Access order is what makes an LRU cache easy to build.
Stacks

A Stack is a linear structure where you add and remove elements from the top only: the most recently added element is the first one removed (LIFO, Last In First Out). Think undo history, a browser's back button, or nested function calls. Using Deque (like ArrayDeque) instead is often recommended for better performance and flexibility.

Stack<String> stack = new Stack<>();
stack.push("Java"); stack.push("Python"); stack.push("C++");
stack.pop();  // removes and returns "C++"
stack.peek(); // returns the top element without removing it
stack.isEmpty(); stack.size();
int position = stack.search("Java"); // 1-based position from the top
int notFound = stack.search("Ruby");  // -1, not found

Conditional statements
int age = 18;
if (age >= 18) System.out.println("You're an adult.");
int age2 = 16;

else System.out.println("Child");
int score = 75;
if (score >= 90) System.out.println("Grade A");
else if (score >= 80) System.out.println("Grade B");
else if (score >= 70) System.out.println("Grade C");
else System.out.println("Fail");
int age3 = 20;
String result = (age3 >= 18) ? "Adult" : "Minor"; // "Adult"

The ternary operator ?: is a compact way to write a simple if-else.

int day = 3;
switch (day) {
  case 1: System.out.println("Monday"); break;
  case 2: System.out.println("Tuesday"); break;
  case 3: System.out.println("Wednesday"); break;
  default: System.out.println("Another day");
}

A switch is efficient for checking a variable against many discrete values. break prevents fall-through into the next case; default is optional and runs when nothing matches.

int day2 = 3;
String result2 = switch (day2) {
  case 1 -> "Monday";
  case 2 -> "Tuesday";
  case 3 -> {
    System.out.println("Logging: mid-week");
    yield "Wednesday";
  }
  case 6, 7 -> "Weekend";
  default -> "Unknown";
};

The enhanced switch (Java 14+) has cleaner syntax, no break needed, and works as an expression that returns a value directly. yield returns a value from a block, useful when more logic is needed first.

Loops
for (int i = 0; i < 5; i++) {
  System.out.println("i = " + i);
}

A for loop is used when you know how many times to repeat something: i = 0 initialization, i < 5 condition, i++ update.

int i = 0;
while (i < 5) {
  System.out.println("i = " + i);
  i++;
}

A while loop runs while a condition holds, without knowing the count ahead of time. It checks the condition before each iteration.

int i2 = 0;
do {
  System.out.println("i = " + i2);
  i2++;
} while (i2 < 5);

A do-while is like while, but always runs at least once since the condition is checked after the first pass.

String[] names = {"Alice","Bob","Charlie"};
for (String name : names) {
  System.out.println(name);
}

The enhanced for-each loop is great for arrays or collections: simpler and cleaner, with no index management, but read-only (elements can't be reassigned directly, unless the element itself is mutable).

for (var p : people) {
  p.age++; // can't reassign the loop variable, but can mutate the object it refers to
}
for (int i3 = 0; i3 < 10; i3++) {
  if (i3 == 5) break;    // exits the loop immediately
}
for (int i4 = 0; i4 < 5; i4++) {
  if (i4 == 2) continue; // skips to the next iteration
}
OOP
Classes and objects
public class Car {
  // Fields (variables that store data)
  String brand;
  int year;

  // Constructor (special method to initialize objects)
  public Car(String brand, int year) {
    this.brand = brand; // "this" refers to the current object
    this.year = year;
  }

  // Method (functions that perform actions)
  public void startEngine() {
    System.out.println(brand + " engine started.");
  }
}

A class is a blueprint for creating objects: it defines the fields (attributes) and methods (behaviors) that instances built from it will have.

Car myCar = new Car("Toyota", 2020); // object of class Car
myCar.startEngine(); // calling a method on the object

An object is an instance of a class, a specific thing built using the class.

Access modifiers
class Counter {
  static int count = 0; // one shared counter
  Counter(){ count++; }
}
Counter a = new Counter();
Counter b = new Counter();
Counter.count; // 2

static makes a field or method belong to the class itself, not to individual instances: a static field is shared by every object, and a static method can be called without creating an object.

public class Car {
  public String brand;
  public void drive(){ System.out.println("Driving..."); }
}

public makes a class, method, constructor, or variable accessible from any other class, regardless of package, useful for anything meant to be globally available across the whole application.

public class BankAccount {
  private double balance;
  public void deposit(double amount){ balance += amount; }
  private void logTransaction(String msg){ System.out.println("LOG: " + msg); }
}

private restricts access to only within the declaring class. It's how encapsulation is enforced, keeping internal details like fields hidden from other classes; commonly used for fields and helper methods that shouldn't be touched from outside.

class PackageClass {
  void display(){ System.out.println("Default access"); }
}

No modifier at all means default (package-private) access: visible only within the same package, not even to subclasses in other packages. Useful for limiting access to a package's internal components without going fully public.

class Animal {
  protected void eat(){ System.out.println("Animal eats"); }
}
class Dog extends Animal {
  public void bark(){
    eat(); // allowed: protected method in the superclass
    System.out.println("Barks");
  }
}

protected allows access within the same package and from subclasses even in other packages. Commonly used in inheritance so derived classes can reuse or override behavior while still keeping unrelated classes out.

Getters and setters

Getters and setters access and update a class's private fields. They're part of encapsulation: keep fields private and expose controlled access instead, in order to protect internal data, control read/write access, add validation logic, and hide implementation details from callers.

public class Person {
  private String name;

  public String getName(){ return name; }

  public void setName(String newName){
    if (newName != null && !newName.isEmpty()) {
      this.name = newName;
    } else {
      System.out.println("Invalid name");
    }
  }
}

p.setName("Alice");
System.out.println(p.getName()); // Alice

Good practice: make fields private, provide public get/set methods, add validation in setters where it matters, and name them consistently as getFieldName()/setFieldName().

Inheritance

Inheritance lets a subclass (child class) inherit fields and methods from a superclass (parent class). It promotes code reuse and models "is-a" relationships. A class can only extend one class (Java supports single inheritance only); super is used to call a parent constructor or access parent members.

class Animal {
  void eat(){ System.out.println("This animal eats food."); }
}
class Dog extends Animal {
  void bark(){ System.out.println("The dog barks."); }
}

Dog myDog = new Dog();
myDog.eat();  // inherited from Animal
myDog.bark(); // defined in Dog
class Animal2 {
  void sound(){ System.out.println("Animal makes a sound."); }
}
class Cat extends Animal2 {
  @Override
  void sound(){ System.out.println("Cat meows."); }
}

Subclasses can override inherited methods with @Override.

super keyword

super refers to the immediate parent class of a subclass, useful anywhere you work with inheritance.

class Animal {
  Animal(String name){ System.out.println("Animal name: " + name); }
}
class Dog extends Animal {
  Dog(){
    super("Buddy"); // calls the Animal constructor
    System.out.println("Dog constructor");
  }
}

super(...) calls a parent constructor. Through Java 23 it had to be the first statement in the subclass constructor; flexible constructor bodies (previewed since Java 22) relax this, and as of Java 25 the feature is final, so super(...) no longer has to come first, as long as earlier statements don't touch the uninitialized instance.

class Animal2 {
  void speak(){ System.out.println("Animal speaks"); }
}
class Dog2 extends Animal2 {
  void speak(){
    super.speak(); // calls Animal's speak()
    System.out.println("Dog barks");
  }
}

If a subclass overrides a method, super.methodName() explicitly calls the parent's version.

class Animal3 {
  String type = "Animal";
}
class Dog3 extends Animal3 {
  String type = "Dog";
  void printType(){
    System.out.println(super.type); // Animal
    System.out.println(this.type);  // Dog
  }
}

If a parent field is hidden by a subclass field of the same name, super.fieldName accesses the parent's version.

Constructor overloading

Constructor overloading means having several constructors in the same class, each with a different parameter list, allowing different ways to initialize an object.

public class Book {
  private String title, author;
  private int year;
  private double price;

  public Book(){ // no parameters
    this.title = "Untitled"; this.author = "Unknown";
  }
  public Book(String title, String author){ // title and author
    this.title = title; this.author = author;
  }
  public Book(String title, String author, int year){ // + year
    this(title, author); this.year = year;
  }
  public Book(String title, String author, int year, double price){ // all fields
    this(title, author, year); this.price = price;
  }
}

Book b1 = new Book();
Book b2 = new Book("1984", "George Orwell");
Book b4 = new Book("Clean Code", "Robert C. Martin", 2008, 35.99);
Method overloading

Method overloading means multiple methods can share a name but have different parameter lists within the same class, giving several ways to call the "same" method. It isn't based on return type alone, you cannot overload by only changing the return type.

public class Calculator {
  public int add(){ return 0; }                          // no parameters
  public int add(int a){ return a; }                     // one int
  public int add(int a, int b){ return a + b; }         // two ints
  public double add(double a, double b){ return a + b; } // two doubles
  public double add(int a, double b){ return a + b; }    // mixed
}

Calculator calc = new Calculator();
calc.add(10, 20);  // two ints
calc.add(5.5, 6.3); // two doubles
Method overriding

Method overriding happens when a subclass provides a new implementation for a method already defined in its superclass, with the same name, parameter list, and a compatible return type, used to customize behavior. @Override tells the compiler "I'm overriding a parent method", catching typos in the name or signature.

class Animal {
  void makeSound(){ System.out.println("Animal makes a sound"); }
}
class Dog extends Animal {
  @Override
  void makeSound(){ System.out.println("Dog barks"); }
}

Animal myDog = new Dog(); // polymorphism
myDog.makeSound(); // "Dog barks": even though myDog is typed as Animal, Dog's version runs at runtime
class Cat extends Animal {
  void makesound(){ // misspelled, this is NOT an override
    System.out.println("Cat meows");
  }
}

Without @Override, a typo like this compiles fine, since it's just treated as a brand-new method rather than an override. Adding @Override to the misspelled version would turn it into a compile-time error.

Abstract classes

An abstract class cannot be instantiated directly; it must be subclassed to be used. Declared with the abstract keyword, it acts as a blueprint for its subclasses, useful when a group of related classes should share a common interface and some behavior, while other parts must be filled in per subclass. It can contain abstract methods (no body), concrete methods, constructors, fields, and static blocks.

abstract class Animal {
  abstract void makeSound(); // abstract method, no body
  void breathe(){ System.out.println("Breathing..."); } // concrete method
}
class Dog extends Animal {
  @Override
  void makeSound(){ System.out.println("Woof!"); }
}

// Animal a = new Animal(); // cannot instantiate an abstract class
Animal dog = new Dog(); // can use the abstract type

Interfaces

An interface is a contract specifying what a class must do, not how. It declares abstract method signatures (plus optionally default or static methods) without implementing the abstract ones; implementing classes must provide concrete bodies for all of them. Since a class can implement several interfaces at once, interfaces are how Java approximates multiple inheritance. They can hold abstract methods (by default), default methods (with a body), static methods, private helper methods, and public static final constants.

interface Animal {
  void makeSound(); // abstract by default
}
class Dog implements Animal {
  @Override
  public void makeSound(){ System.out.println("Woof!"); }
}
interface Flyable { void fly(); }
interface Swimmable { void swim(); }
class Duck implements Flyable, Swimmable {
  public void fly(){ System.out.println("Duck flies."); }
  public void swim(){ System.out.println("Duck swims."); }
}

Sealed classes

A sealed class (Java 17+) restricts which other classes may extend or implement it, giving stronger encapsulation, safer and more predictable inheritance, clearer class hierarchies, and better support for pattern matching (Java 21+). Before sealed classes, anyone could extend an open class, making it hard to reason about every possible subclass, especially for pattern matching or API design.

public sealed class Shape permits Circle, Rectangle, Triangle {}

A sealed class must list which classes may extend it via permits; any other attempt to extend it is a compile-time error.

// final: no one can extend it further
public final class Circle extends Shape {}
// sealed: can still restrict its own subclasses
public sealed class Rectangle extends Shape permits Square {}
// non-sealed: opens back up, anyone can extend it
public non-sealed class Triangle extends Shape {}

Every direct subclass of a sealed class must be marked final, sealed, or non-sealed, giving fine-grained control over further extension.

String describe(Shape s){
  return switch (s) {
    case Circle c -> "Circle";
    case Square sq -> "Square";
    case Triangle t -> "Triangle";
  };
}

Because the compiler knows every possible subclass of a sealed hierarchy, a switch over them can be exhaustive with no default needed.

Records

A record (Java 16+) is a special class designed to hold immutable data: Java generates the constructor, accessors, equals(), hashCode(), and toString() automatically. Think of it as a concise, built-in way to write data-carrier classes (sometimes called DTOs or value objects).

// Old way: lots of boilerplate
public class Person {
  private final String name;
  private final int age;
  public Person(String name, int age){ this.name=name; this.age=age; }
  public String name(){ return name; }
  public int age(){ return age; }
  // equals(), hashCode(), toString() all hand-written...
}

// With records:
public record Person(String name, int age) {}

A record automatically gets a canonical constructor, read-only accessor methods matching the field names (name(), age(), not getName()), equals()/hashCode() based on all its components, a toString() like Person[name=Alice, age=30], and fields that are final (records are immutable by default).

public record Point(int x, int y) implements Drawable {}

Records can implement interfaces, but can't add mutable fields or setters, and can't extend another class (they implicitly extend java.lang.Record).

public record Person(String name, int age) {
  public String greet(){ return "Hello, I'm " + name; } // custom logic is allowed
}
public record Person2(String name, int age) {
  public Person2 { // compact constructor, for validation
    if (age < 0) throw new IllegalArgumentException("Age cannot be negative");
  }
}

A compact constructor lets you add validation; Java still assigns the fields for you unless you write out the full canonical constructor yourself.

  void makeSound(){ System.out.println("Generic sound"); }
}

  void bark(){ System.out.println("Bark!"); }
}

Inheritance ("is-a"): a subclass inherits fields and methods from a superclass. Useful when one class is a more specific version of another, but it's tight coupling, Java only allows a single parent class, and it's not a fit for every relationship.

  String name;
  void diagnose(Patient patient){
    System.out.println("Doctor " + name + " is diagnosing " + patient.name + ".");
  }
}

Association ("uses-a"): one object uses or connects to another without implying ownership, usually by holding a reference. It can be one-directional (only one side knows about the other) or bidirectional.

class Car {
  Engine engine; // aggregation
  Car(Engine engine){ this.engine = engine; }
}

Aggregation ("has-a", weak ownership): a class holds a reference to another whose lifetime is independent of the owner. An Engine can be shared across multiple Cars, and can outlive any one of them.

class Human {
  private Heart heart = new Heart(); // composition
  void live(){ heart.beat(); }
}

Composition ("part-of", strong ownership): a class creates and controls the lifecycle of another internally; the part cannot exist independently. If the Human is destroyed, its Heart goes with it, a stronger coupling than aggregation.

Type casting

Type casting converts a variable from one data type to another: widening (implicit) casting, or narrowing (explicit) casting.

String s = "123";

Converting between non-compatible types, like String to int, requires parsing rather than casting.

List<String> result2 = names.parallelStream().map(String::toUpperCase).collect(Collectors.toList());

Parallel streams can use multiple cores, but aren't always faster, best for large, CPU-bound tasks.

Optional class

Optional<T> (Java 8+) is a container that may or may not hold a non-null value: instead of returning null for a missing value, it gives you an explicit, safe wrapper.

Optional<String> opt1 = Optional.of("Hello");     // value present
Optional<String> opt2 = Optional.empty();          // no value
Optional<String> opt3 = Optional.ofNullable(null); // may or may not be null

of() throws NullPointerException if given null; ofNullable() safely produces an empty Optional instead; empty() creates an empty one directly.

if (name.isPresent()) {
  System.out.println("Value: " + name.get());
}

Calling .get() directly is discouraged unless you're sure a value is present; the safer alternatives below are usually better.

Optional.ofNullable(null).orElse("Default");                        // fallback value
Optional.ofNullable(null).orElseGet(() -> "Computed Default");      // fallback via a Supplier, computed only if needed
Optional.ofNullable(null).orElseThrow(() -> new IllegalArgumentException()); // throws if missing
Optional<String> name2 = Optional.of("java");
Optional<String> upper = name2.map(String::toUpperCase); // transforms the value if present
Optional<String> upper2 = name2.flatMap(n -> Optional.of(n.toUpperCase()));

flatMap() is used when the mapping function itself returns an Optional.

name2 = name2.filter(n -> n.startsWith("J")); // keeps the value only if it matches a condition
// Without Optional:
if (person != null && person.getAddress() != null) {
  return person.getAddress().getCity();
} else return "Unknown";

// With Optional:
String city = Optional.ofNullable(person)
  .map(Person::getAddress)
  .map(Address::getCity)
  .orElse("Unknown");

Chaining Optional avoids nested null checks entirely, no NullPointerException, no messy conditionals.

Method references

A method reference is shorthand for a lambda that just calls an existing method.

// Without method reference:
list.forEach(name -> System.out.println(name));
// With method reference:
list.forEach(System.out::println);
numbers.sort(Integer::compare); // reference to a static method: ClassName::staticMethodName
Printer printer = new Printer();
names.forEach(printer::printName); // reference to an instance method of a particular object: instance::instanceMethodName
names.sort(String::compareToIgnoreCase); // reference to an instance method of an arbitrary object: ClassName::instanceMethodName
Supplier<StringBuilder> builderSupplier = StringBuilder::new; // reference to a constructor: ClassName::new
names.stream()
  .map(String::toUpperCase)
  .sorted(String::compareTo)
  .forEach(System.out::println);

Method references combine naturally with Streams.

var (Java 10+)
// Without var:
String message = "Hello, Java!";
// With var:
var message2 = "Hello, Java!";

var (Java 10+) lets the compiler infer a local variable's type from context; Java remains statically typed, the variable behaves exactly as if the type had been written explicitly.

// var name; // error, type can't be inferred without initialization
var name = "Java"; // OK

A var variable must be initialized immediately.

class Example {
  // var field = 10; // not allowed
}

var can't be used for fields or method parameters, only for local variables inside methods, constructors, or blocks.

// var data = null; // compiler can't infer a type from null alone
var num = 42;        // int
var pi = 3.14;       // double
var isReady = true; // boolean
var text = "Hello";  // String
var list = new ArrayList<String>(); // ArrayList<String>

  System.out.println(entry.getKey() + " -> " + entry.getValue());
}

var is handy for reducing redundancy when the type is obvious, e.g. in loops and streams, but is best avoided when the inferred type isn't clear, since that hurts readability.

Concurrency
Concurrency & Multithreading

Multithreading is the most common way to achieve concurrency in Java. A thread is a lightweight unit of execution: a process can run several threads, each independent, but all sharing the same heap (so they can access shared objects) while keeping separate call stacks. One program might use one thread for user input and another for background work.

class MyThread extends Thread {
  public void run(){ System.out.println("Thread running: " + Thread.currentThread().getName()); }
}
new MyThread().start(); // never call run() directly, that runs on the current thread
class MyRunnable implements Runnable {
  public void run(){ System.out.println("Running in thread: " + Thread.currentThread().getName()); }
}
new Thread(new MyRunnable()).start();

Implementing Runnable is generally preferred over extending Thread, since it leaves the class free to extend something else.

Thread t1 = new Thread(() -> System.out.println("Running in thread: " + Thread.currentThread().getName()));
t1.start();

A lambda (Java 8+) is the cleanest, most common way to start a thread today.

A thread moves through several states: NEW (created, not started), RUNNABLE (ready, waiting for CPU), RUNNING (executing), BLOCKED/WAITING (waiting on a resource or another thread), TERMINATED (finished).

Thread worker = new Thread(() -> {
  try { Thread.sleep(1000); } catch (InterruptedException ignored) {}
  System.out.println("Worker thread done!");
});
worker.start();
worker.join(); // current thread waits until 'worker' finishes

Thread.sleep(ms) pauses the current thread without releasing any locks it holds, and can throw InterruptedException. join() makes the caller wait for another thread to finish, commonly used to ensure background work completes before the program exits.

t1.interrupt(); // signals a thread to stop; if it's sleeping/waiting, it throws InterruptedException
t1.isAlive();   // true once started, false before starting or after finishing
class Counter {
  private int count = 0;
  public synchronized void increment(){ count++; }
}

Since threads share memory, multiple threads modifying the same data at once causes race conditions. synchronized ensures only one thread at a time can run increment().

private static volatile boolean running = true;

volatile ensures a variable shared across threads is always read from main memory, so changes are visible immediately, useful for a simple shared flag (like running) where you need visibility but not a compound operation like ++.

ExecutorService executor = Executors.newFixedThreadPool(3); // pool of 3 reusable threads
for (int i = 1; i <= 5; i++) {
  int taskId = i;
  executor.submit(() -> System.out.println("Running task " + taskId));
}
executor.shutdown(); // stops accepting new tasks once submitted ones finish

An ExecutorService manages thread creation, reuse, and shutdown for you, instead of creating threads manually.

Callable<Integer> task = () -> { Thread.sleep(1000); return 42; };
Future<Integer> result = executor.submit(task);
result.get(); // blocks until the task completes, returns 42

A Callable is like a Runnable but returns a value (and can throw exceptions); the Future represents that pending result, and can also be polled with isDone() or cancelled with cancel(true).

CountDownLatch latch = new CountDownLatch(3);
for (int i = 1; i <= 3; i++) {
  new Thread(() -> { /* work */ latch.countDown(); }).start();
}
latch.await(); // blocks until the count reaches zero

CountDownLatch lets one or more threads wait until a set number of other threads finish.

Semaphore semaphore = new Semaphore(2); // only 2 permits available at once
semaphore.acquire(); // blocks if no permit is free
try { /* work */ } finally { semaphore.release(); }

A Semaphore limits how many threads can access a resource simultaneously, useful for capping something like database connections.

ReentrantLock lock = new ReentrantLock();
if (lock.tryLock()) {
  try { /* work */ } finally { lock.unlock(); } // always unlock in a finally block
}

ReentrantLock is a more flexible alternative to synchronized: explicit lock/unlock, fairness options, and non-blocking tryLock().

Object lock2 = new Object();
synchronized (lock2) {
  lock2.wait(); // releases the lock and pauses until notified
}
// on another thread:
synchronized (lock2) { lock2.notify(); }    // wakes one waiting thread
synchronized (lock2) { lock2.notifyAll(); } // wakes every waiting thread

wait()/notify()/notifyAll() are the low-level coordination primitives behind higher-level tools like locks and latches.

Parallelism

Parallelism happens when multiple threads run on multiple CPU cores at the same time. The number of available cores can be checked at runtime.

int cores = Runtime.getRuntime().availableProcessors();
System.out.println("Available cores: " + cores);
ExecutorService pool = Executors.newFixedThreadPool(4);
for (int i = 0; i < 4; i++) {
  pool.submit(() -> System.out.println(Thread.currentThread().getName() + " working"));
}
pool.shutdown();

A fixed-size ExecutorService thread pool: on a machine with 4+ cores, these tasks can genuinely run at the same time, not just interleaved.

List<Integer> numbers = List.of(1,2,3,4,5);
int sum = numbers.parallelStream()
  .mapToInt(Integer::intValue)
  .sum();

Parallel streams are the simplest way to get parallelism: Java automatically splits the work across threads using a shared ForkJoinPool behind the scenes.

class SumTask extends RecursiveTask<Long> {
  private final long[] numbers; private final int start, end;
  SumTask(long[] numbers, int start, int end){
    this.numbers=numbers; this.start=start; this.end=end;
  }
  protected Long compute(){
    if (end - start <= 1000) {
      long sum = 0;
      for (int i=start;i<end;i++) sum += numbers[i];
      return sum;
    }
    int mid = (start+end)/2;
    SumTask left = new SumTask(numbers, start, mid);
    SumTask right = new SumTask(numbers, mid, end);
    left.fork();
    return right.compute() + left.join();
  }
}

For recursive tasks (splitting a big job into smaller ones), the Fork/Join framework can be used directly; it's what parallelStream() is built on top of.

Advanced
Generics

Generics (Java 5+) let a class, interface, or method work with any type while still keeping compile-time type safety, eliminating manual casting and catching type mismatches before runtime.

// Without generics: mixing types compiles, but crashes at runtime
List list = new ArrayList();
list.add("Java"); list.add(42);
String s = (String) list.get(1); // ClassCastException at runtime

// With generics: the compiler catches the mistake immediately
List<String> names = new ArrayList<>();
names.add("Java");
// names.add(42); // won't even compile
class Box<T> { // T is a type parameter
  private T value;
  public void set(T value){ this.value = value; }
  public T get(){ return value; }
}
public static <T> void printTwice(T value){
  System.out.println(value);
  System.out.println(value);
}

Methods can be generic too, even inside a non-generic class; the type is usually inferred automatically at the call site.

class Calculator<T extends Number> {
  public double add(T a, T b){ return a.doubleValue() + b.doubleValue(); }
}
// Calculator would be invalid: String doesn't extend Number

Bounds (T extends Number) restrict which types a generic parameter accepts.

static void printList(List<?> list){
  for (Object o : list) System.out.println(o);
}

Wildcards: ? any type, ? extends T any subtype of T, ? super T any supertype of T.

class Response<T> {
  boolean success; String message; T data;
  Response(boolean success, String message, T data){
    this.success=success; this.message=message; this.data=data;
  }
}
Response<User> userResponse = new Response<>(true, "User fetched", new User());

A common real use: instead of writing a separate Response class per endpoint (User, Product, Order...), one generic Response<T> adapts to any of them with full compile-time safety.

Annotations

An annotation is metadata attached to code with an @ symbol: it doesn't change what the code does directly, but the compiler, tools, or frameworks can read it at compile time or runtime.

Custom annotations are defined with @interface. This is the same mechanism frameworks like Spring and JUnit build on: they scan for annotated methods with reflection and act on them.

Reflection

Reflection lets a program inspect and manipulate classes, methods, fields, and constructors at runtime, even without knowing their names at compile time, which is how frameworks like Spring, JUnit, and Hibernate work under the hood.

Class<?> clazz = Class.forName("java.util.ArrayList"); // loads the class at runtime
for (Method m : clazz.getDeclaredMethods()) {
  System.out.println(m.getName());
}

Similar calls exist for getDeclaredFields() and getDeclaredConstructors().

class Person {
  private String name;
  Person(String name){ this.name = name; }
}

Person p = new Person("Alice");
Field field = Person.class.getDeclaredField("name");
field.setAccessible(true); // bypasses the private access check
field.set(p, "Bob");
System.out.println(field.get(p)); // "Bob"

setAccessible(true) lets reflection read or modify private fields and invoke private methods, things that are normally impossible from outside the class.

Enums

An enum is a special type representing a fixed set of constants; internally, enums are classes with fields, constructors (always private), and methods. Good fits: days of the week, directions, status codes, user roles. Not a good fit: dynamic values that change at runtime.

enum Day { MONDAY, TUESDAY, WEDNESDAY, THURSDAY, FRIDAY, SATURDAY, SUNDAY }
Day today = Day.MONDAY;
if (today == Day.MONDAY) System.out.println("Start of week");
enum Status {
  SUCCESS(200), NOT_FOUND(404), SERVER_ERROR(500);
  private final int code;
  Status(int code){ this.code = code; }
  int getCode(){ return code; }
}

Enum constants can carry their own fields, set through a private constructor.

enum Operation {
  ADD { public double apply(double a, double b){ return a+b; } },
  MULTIPLY { public double apply(double a, double b){ return a*b; } };
  public abstract double apply(double a, double b);
}
Day.values();          // array of all constants
Day.valueOf("MONDAY");  // string -> constant
Day.FRIDAY.name();     // "FRIDAY"
Day.MONDAY.ordinal();  // 0, its index

switch (today) {
  case MONDAY -> System.out.println("Back to work!");
  case FRIDAY -> System.out.println("Almost weekend!");
  default -> System.out.println("Some other day");
}
interface Printable { void print(); }
enum Color implements Printable {
  RED, GREEN, BLUE;
  public void print(){ System.out.println(this.name()); }
}

Enums cannot extend a class (they already implicitly extend Enum), but they can implement interfaces.

Inner/nested classes
class Outer {
  static class Nested {
    void print(){ System.out.println("Inside static nested class"); }
  }
}
new Outer.Nested().print();

Static nested classes don't need an instance of the outer class, and cannot access the outer class's instance members directly.

class Outer {
  private String message = "Hello from Outer";
  class Inner {
    void print(){ System.out.println(message); } // can access outer fields
  }
}
Outer outer = new Outer();
Outer.Inner inner = outer.new Inner();

A non-static (member) inner class can access all of the outer class's members, including private ones, but must be created from an outer instance.

class Outer {
  void doSomething(){
    class LocalInner {
      void print(){ System.out.println("I'm inside a method!"); }
    }
    new LocalInner().print();
  }
}

A local inner class is scoped to a single method, often for a small helper; it can access final or effectively-final local variables.

Runnable r = new Runnable(){
  @Override
  public void run(){ System.out.println("Running in anonymous inner class!"); }
};
new Thread(r).start();

Anonymous inner classes have no name, often used for Runnable tasks, event handlers, or a quick interface implementation. Modern Java often reaches for a lambda instead.

Java modules (JPMS)

(Java 9+) The Java Platform Module System groups related packages into named, strongly-encapsulated modules: you expose only what you explicitly choose, instead of every public class in a package being visible to everyone.

module my.module {
  requires other.module;
  exports com.example.api;
}

The module-info.java descriptor names the module, lists its dependencies, and controls which of its packages are exposed. Packages that aren't exported are completely hidden, as if they don't exist to other modules.

requires java.sql;
requires utils.module;
opens com.example.model; // allows runtime reflection on this package
requires transitive other.module; // anyone requiring this module also gets other.module

uses com.example.PaymentProcessor;

  with com.example.StripeProcessor;

uses/provides ... with support the ServiceLoader mechanism, enabling a modular, pluggable application architecture.

Timers

The classic Timer/TimerTask runs on a single thread, so a long-running task blocks every other scheduled task. Not ideal for modern applications.

ScheduledExecutorService scheduler = Executors.newScheduledThreadPool(1);
scheduler.scheduleAtFixedRate(
  () -> System.out.println("Tick"), 0, 2, TimeUnit.SECONDS);

ScheduledExecutorService is the modern replacement: it's backed by a thread pool, so it can run multiple tasks without one blocking another, and gives more predictable timing.
