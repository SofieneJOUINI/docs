Clean Code Principles

🔴 Variables naming is the cheapest form of documentation

Use meaningful variable names

Distinguish names so the reader immediately understands what differentiates them. Don't rely on near-identical names like a1/a2/a3 to carry meaning.

Bad
function between<T>(a1: T, a2: T, a3: T): boolean {
  return a2 <= a1 && a1 <= a3;
}
Good
function between<T>(value: T, left: T, right: T): boolean {
  return left <= value && value <= right;
}

Use pronounceable variable names

If you can't say it out loud, you can't discuss it without sounding ridiculous.

Bad
type DtaRcrd102 = {
  genymdhms: Date;
  modymdhms: Date;
  pszqint: number;
}
Good
type Customer = {
  generationTimestamp: Date;
  modificationTimestamp: Date;
  recordId: number;
}

Use the same vocabulary for the same type of variable

Pick one word for one concept and stick to it everywhere.

Bad
function getUserInfo(): User;
function getUserDetails(): User;
function getUserData(): User;
Good
function getUser(): User;

Use searchable names

We read far more code than we write, so names must be searchable, not just readable. Unnamed "magic" numbers/strings hurt anyone trying to grep or understand the codebase. A linter can help flag them.

Bad
// What the heck is 86400000 for?
setTimeout(restart, 86400000);
Good
const MILLISECONDS_PER_DAY = 24 * 60 * 60 * 1000;
setTimeout(restart, MILLISECONDS_PER_DAY);

Use explanatory variables

Name the intermediate pieces you're working with instead of leaving the reader to infer them.

Bad
for (const keyValue of users) {
  // iterate through users map
}
Good
for (const [id, user] of users) {
  // iterate through users map
}

Avoid Mental Mapping

Explicit beats implicit: clarity is king. Don't force the reader to remember that u means "user".

Bad
const u = getUser();
const s = getSubscription();
const t = charge(u, s);
Good
const user = getUser();
const subscription = getSubscription();
const transaction = charge(user, subscription);

Don't add unneeded context

If the containing type/class already tells you something, don't repeat it in every field name.

Bad
type Car = {
  carMake: string;
  carModel: string;
  carColor: string;
}
Good
type Car = {
  make: string;
  model: string;
  color: string;
}

Use default arguments instead of short circuiting or conditionals

Default parameters are usually cleaner than a manual fallback check.

Bad
function loadPages(count?: number) {
  const loadCount = count !== undefined ? count : 10;
}
Good
function loadPages(count: number = 10) {
  // ...
}

Use enum to document the intent

Enums help communicate that what matters is the values being different from each other, not their exact underlying value.

Bad
const GENRE = {
  ROMANTIC: 'romantic',
  DRAMA: 'drama',
  COMEDY: 'comedy',
};
projector.configureFilm(GENRE.COMEDY);
Good
enum GENRE {
  ROMANTIC,
  DRAMA,
  COMEDY,
}
projector.configureFilm(GENRE.COMEDY);
🔴 Functions small, single-purpose, honestly named

Function arguments (2 or fewer ideally)

Fewer parameters means easier testing: beyond two or three you get a combinatorial explosion of cases to cover. If you truly need many inputs, pass an object and destructure it. Destructuring gives you: instant clarity on what the function uses, a way to simulate named parameters, cloning of primitive values passed in (helps avoid side effects; note nested objects/arrays are not cloned), and unused-property warnings from TypeScript.

Bad
function createMenu(title: string, body: string,
  buttonText: string, cancellable: boolean) { /* ... */ }

createMenu('Foo', 'Bar', 'Baz', true);
Good
type MenuOptions = { title: string; body: string;
  buttonText: string; cancellable: boolean };

function createMenu(options: MenuOptions) { /* ... */ }

createMenu({ title: 'Foo', body: 'Bar',
  buttonText: 'Baz', cancellable: true });

Functions should do one thing

Arguably the single most important rule here. Functions that only do one thing are easier to compose, test, and reason about, and they refactor cleanly.

Bad
function emailActiveClients(clients: Client[]) {
  clients.forEach((client) => {
    const record = database.lookup(client);
    if (record.isActive()) email(client);
  });
}
Good
function emailActiveClients(clients: Client[]) {
  clients.filter(isActiveClient).forEach(email);
}
function isActiveClient(client: Client) {
  const record = database.lookup(client);
  return record.isActive();
}

Function names should say what they do

Bad
function addToDate(date: Date, month: number): Date { /* ... */ }
// unclear what's being added
addToDate(date, 1);
Good
function addMonthToDate(date: Date, month: number): Date { /* ... */ }
addMonthToDate(date, 1);

Functions should only be one level of abstraction

Mixing high-level orchestration with low-level detail in the same function is a sign it's doing too much. Splitting by abstraction level improves reusability and testability.

Example: a parseCode() that tokenizes, lexes, and parses in one giant function should be split into tokenize(), then a separate parse step, each handling one level of abstraction.

Remove duplicate code

Duplicate logic means every future change has to happen in multiple places, and it is easy to forget one. When two things are almost-but-not-quite identical, extract a shared abstraction (a common parent, a union type, a shared method) rather than keeping parallel copies.

Caveat: getting the abstraction right matters. A bad abstraction can be worse than the duplication it replaces, and it introduces an indirect dependency between the two places that use it. If two similar-looking implementations actually live in different domains, duplication may be the right call.

Set default objects with Object.assign or destructuring

Manually checking config.title = config.title || 'Foo' for every field is noisy. Prefer Object.assign, the spread operator, or destructured defaults.

Bad
function createMenu(config: MenuConfig) {
  config.title = config.title || 'Foo';
  config.body = config.body || 'Bar';
  // ...
}
Good
function createMenu({ title = 'Foo', body = 'Bar',
  buttonText = 'Baz', cancellable = true }: MenuConfig) {
  // ...
}

Note: spread and Object.assign() aren't quite identical: spread defines new properties, Object.assign() sets them. Enable --strictNullChecks so passing explicit undefined/null is caught at compile time.

Don't use flags as function parameters

A boolean flag tells the reader the function branches into two different behaviors, meaning it's doing more than one thing. Split it into two functions instead.

Bad
function createFile(name: string, temp: boolean) {
  if (temp) fs.create(`./temp/${name}`);
  else fs.create(name);
}
Good
function createTempFile(name: string) {
  createFile(`./temp/${name}`);
}
function createFile(name: string) {
  fs.create(name);
}

Avoid Side Effects (part 1)

A function has a side effect if it does anything besides taking a value in and returning a value out: writing a file, mutating a global, etc. Side effects are sometimes necessary, but centralize where they happen, in one dedicated service, not scattered across many functions.

Bad
let name = 'Robert C. Martin';
function toBase64() {
  name = btoa(name); // mutates the outer variable
}
toBase64();
console.log(name); // not what you'd expect
Good
const name = 'Robert C. Martin';
function toBase64(text: string): string {
  return btoa(text);
}
const encodedName = toBase64(name);
console.log(name); // unchanged

Avoid Side Effects (part 2)

Objects and arrays are mutable in JS/TS, so passing them into a function risks unexpected shared-state bugs, e.g. a shopping cart array mutated mid-request by an unrelated click. Prefer cloning-and-returning over in-place mutation.

Bad
function addItemToCart(cart: CartItem[], item: Item): void {
  cart.push({ item, date: Date.now() });
}
Good
function addItemToCart(cart: CartItem[], item: Item): CartItem[] {
  return [...cart, { item, date: Date.now() }];
}

Two caveats: sometimes mutating the input really is what you want (rare in practice, see "pure function"); and cloning large objects has a real performance cost, though good libraries make this cheap in practice.

Don't write to global functions

Extending built-ins like Array.prototype risks silently clashing with another library doing the same thing. Extend a class instead of touching the global.

Bad
Array.prototype.diff = function<T>(other: T[]): T[] {
  const hash = new Set(other);
  return this.filter(elem => !hash.has(elem));
};
Good
class MyArray<T> extends Array<T> {
  diff(other: T[]): T[] {
    const hash = new Set(other);
    return this.filter(elem => !hash.has(elem));
  }
}

Favor functional programming over imperative programming

Bad
let totalOutput = 0;
for (let i = 0; i < contributions.length; i++) {
  totalOutput += contributions[i].linesOfCode;
}
Good
const totalOutput = contributions
  .reduce((total, c) => total + c.linesOfCode, 0);

Encapsulate conditionals

Give a compound boolean expression a name so the intent reads clearly at the call site.

Bad
if (subscription.isTrial || account.balance > 0) { /* ... */ }
Good
function canActivateService(subscription, account) {
  return subscription.isTrial || account.balance > 0;
}
if (canActivateService(subscription, account)) { /* ... */ }

Avoid negative conditionals

Bad
function isEmailNotUsed(email: string): boolean { /* ... */ }
if (isEmailNotUsed(email)) { /* ... */ }
Good
function isEmailUsed(email: string): boolean { /* ... */ }
if (!isEmailUsed(email)) { /* ... */ }

Avoid conditionals

Sounds impossible, but polymorphism can often replace branching entirely. A function/class full of if/switch branches is usually a sign it's doing more than one thing.

Classic example: an Airplane.getCruisingAltitude() with a switch on airplane type becomes cleaner as an abstract Airplane base class with one subclass per type, each implementing its own getCruisingAltitude().

Avoid type checking

TypeScript's whole value proposition is static typing: use it. Prefer well-typed parameters/returns over runtime instanceof branching; it also makes refactoring easier.

Bad
function travelToTexas(vehicle: Bicycle | Car) {
  if (vehicle instanceof Bicycle) vehicle.pedal(/*...*/);
  else if (vehicle instanceof Car) vehicle.drive(/*...*/);
}
Good
type Vehicle = Bicycle | Car;
function travelToTexas(vehicle: Vehicle) {
  vehicle.move(currentLocation, new Location('texas'));
}

Don't over-optimize

Modern engines already optimize plenty under the hood, so hand-optimizing prematurely often just wastes time and adds noise. Optimize based on measured bottlenecks, not guesses.

Remove dead code

Unused code is as harmful as duplicate code: delete it. Version control keeps the history if you ever need it back.

Use iterators and generators

Generators/iterables are a great fit for stream-like data: they decouple the caller from how items are produced, execute lazily (items are produced on demand), support native for-of, and enable optimized iterator patterns. Compare building an entire array up front vs. yielding values one at a time.

Bad
function fibonacci(n: number): number[] {
  const items = [0, 1];
  while (items.length < n) {
    items.push(items[items.length-2] + items[items.length-1]);
  }
  return items;
}
Good
function* fibonacci(): IterableIterator<number> {
  let [a, b] = [0, 1];
  while (true) {
    yield a;
    [a, b] = [b, a + b];
  }
}
🟢 Objects & Data Structures encapsulation and immutability

Use getters and setters

Accessing data through getters/setters (instead of raw public fields) means: you can add behavior later without touching every call site, validation on set is trivial to add, the internal representation stays encapsulated, logging/error-handling on access is easy to bolt on, and properties can be lazily loaded.

Bad
type BankAccount = { balance: number };
const account: BankAccount = { balance: 0 };
if (value < 0) throw new Error('Cannot set negative balance.');
account.balance = value;
Good
class BankAccount {
  private accountBalance = 0;
  get balance() { return this.accountBalance; }
  set balance(value: number) {
    if (value < 0) throw new Error('Cannot set negative balance.');
    this.accountBalance = value;
  }
}

Make objects have private/protected members

TypeScript supports public (default), protected, and private access modifiers, use them to hide implementation details.

Bad
class Circle {
  radius: number;
  constructor(radius: number) { this.radius = radius; }
}
Good
class Circle {
  constructor(private readonly radius: number) {}
}

Prefer immutability

Mark properties readonly, use ReadonlyArray<T> for arrays that shouldn't be mutated, and prefer as const assertions for literal values that should stay frozen.

Bad
const config = { hello: 'world' };
config.hello = 'world'; // silently allowed, value changed

const array = [1, 3, 5];
array[0] = 10; // silently allowed
Good
const config = { hello: 'world' } as const;
config.hello = 'world'; // compile error

const array = [1, 3, 5] as const;
array[0] = 10; // compile error

type vs. interface

Use type when you need a union or intersection. Use interface when you want extends/implements. No hard rule: use whichever fits the situation.

Bad
interface EmailConfig { /* ... */ }
interface DbConfig { /* ... */ }
interface Config { /* ... */ } // can't express a union here
Good
type EmailConfig = { /* ... */ };
type DbConfig = { /* ... */ };
type Config = EmailConfig | DbConfig;

interface Shape { /* ... */ }
class Circle implements Shape { /* ... */ }
🟢 Classes small, cohesive, composed

Classes should be small

Class size is measured by responsibility, not line count. Following Single Responsibility, a class with a long, unrelated list of methods (settings, progress, users, navigation, versioning, all in one Dashboard class) should be split so each piece owns one concern.

High cohesion and low coupling

Cohesion = how related a class's members are to each other; ideally every field is used by every method (rarely fully achievable, but aim high). Coupling = how dependent two classes are on each other; low coupling means a change in one doesn't ripple into the other. Good design has high cohesion, low coupling.

Warning sign: a class like UserManager where each private field is only used by one subset of its methods (DB access here, email sending there) is really two responsibilities glued together, so split it into e.g. UserService and UserNotifier.

Prefer composition over inheritance

A classic Gang-of-Four maxim. When your instinct reaches for inheritance, check whether composition models the problem better. Inheritance tends to make more sense when:

  • The relationship is genuinely "is-a", not "has-a" (Human→Animal, not User→UserDetails)
  • You can meaningfully reuse code from the base class
  • You want a change in the base class to ripple globally through all derived classes

Example: EmployeeTaxData extends Employee is wrong: an employee has tax data, doesn't is-a tax data. Model it as composition: Employee holds a reference to an EmployeeTaxData instance instead.

Use method chaining

Returning this from setter-style methods enables fluent, expressive call chains and less verbose call sites, a pattern widely used across libraries (e.g. query builders).

Bad
queryBuilder.from('users');
queryBuilder.page(1, 100);
queryBuilder.orderBy('firstName', 'lastName');
const query = queryBuilder.build();
Good
const query = new QueryBuilder()
  .from('users')
  .page(1, 100)
  .orderBy('firstName', 'lastName')
  .build();
🟣 SOLID five design principles for maintainable OO code

Single Responsibility Principle (SRP)

"There should never be more than one reason for a class to change." (Robert C. Martin, Clean Code)

Packing a class with too much functionality means it lacks conceptual cohesion and gains many reasons to change, making it hard to predict how a modification ripples to dependents.

Example: a UserSettings class that both changes settings and verifies credentials should split credential verification into its own UserAuth class.

Open/Closed Principle (OCP)

"Software entities should be open for extension, but closed for modification." (Bertrand Meyer)

Users should be able to add new functionality without changing existing, already-tested code.

Example: instead of an HttpRequester that branches on instanceof AjaxAdapter vs instanceof NodeAdapter, define an abstract Adapter with a request() method, so new adapter types can be added without touching HttpRequester at all.

Liskov Substitution Principle (LSP)

"If S is a subtype of T, objects of type T may be replaced with objects of type S without altering any of the desirable properties of the program." (Barbara Liskov)

In plain terms: a subclass should be swappable in for its parent class without breaking correctness. The classic trap is Square extending Rectangle: mathematically a square is a rectangle, but modeling it via inheritance breaks LSP the moment setWidth/setHeight behave differently on Square than expected.

Fix: don't force Square to extend Rectangle. Give both an abstract Shape base with its own getArea() implementation instead, so there is no surprising behavior when swapping one shape for another.

Interface Segregation Principle (ISP)

"Clients should not be forced to depend upon interfaces that they do not use."

Closely related to SRP: design abstractions so a consumer only has to implement/depend on what it actually needs, rather than a bloated "does everything" interface.

Example: a single SmartPrinter interface with print()/fax()/scan() forces an EconomicPrinter to implement fax/scan methods it can't support (throwing errors instead). Split into separate Printer, Fax, and Scanner interfaces so each class only implements what it supports.

Dependency Inversion Principle (DIP)

  1. High-level modules should not depend on low-level modules; both should depend on abstractions.
  2. Abstractions should not depend on details; details should depend on abstractions.

Usually achieved through Dependency Injection / an Inversion of Control container. This decouples modules from each other, which is what makes refactoring safe.

Example: a ReportReader that directly instantiates new XmlFormatter() is locked into XML forever. Depend on a Formatter interface instead, and inject whichever concrete formatter (XmlFormatter, JsonFormatter) is needed.
🟠 Testing tests are part of the product, not an afterthought

Why it matters

Without adequate tests, every deploy is a gamble. Aiming for 100% coverage (statements and branches) builds real confidence, but that requires a good testing framework and a good coverage tool. Whatever your team picks, write tests for every new feature/module and hit your coverage bar before shipping or refactoring.

The three laws of TDD

  1. You may not write production code until you have a failing unit test.
  2. You may not write more of a test than is sufficient to fail; compilation failures count as failures.
  3. You may not write more production code than is sufficient to pass the one currently-failing test.

F.I.R.S.T. rules

LetterMeaning
FastTests should run fast, since you'll be running them constantly
IndependentTests shouldn't depend on each other: the same result whether run alone or together, in any order
RepeatableShould pass reliably in any environment, no excuses for flakiness
Self-ValidatingA test should output a plain Pass/Fail, with no manually diffing log files
TimelyWrite the test before the production code; writing it after tends to be much harder

Single concept per test

Tests should follow SRP too: one assertion (or one concept) per test, so a failure immediately tells you exactly what broke.

Bad
it('handles date boundaries', () => {
  assert.equal('1/31/2015', new AwesomeDate('1/1/2015').addDays(30));
  assert.equal('2/29/2016', new AwesomeDate('2/1/2016').addDays(28));
  assert.equal('3/1/2015', new AwesomeDate('2/1/2015').addDays(28));
});
Good
it('handles 30-day months', () => { /* one assert */ });
it('handles leap year', () => { /* one assert */ });
it('handles non-leap year', () => { /* one assert */ });

The name of the test should reveal its intention

When a test fails, its name is the very first clue about what went wrong, so make it descriptive, not cryptic.

Bad
it('2/29/2020', () => { /* ... */ });
it('throws', () => { /* ... */ });
Good
it('should handle leap year', () => { /* ... */ });
it('should throw when format is invalid', () => { /* ... */ });
🟠 Concurrency promises, async/await

Prefer promises vs callbacks

Callbacks nest badly (the classic "callback hell") and are hard to read. Promisify callback-style APIs instead (util.promisify in Node, or libraries like pify).

Bad
get(url, (error, response) => {
  if (error) callback(error);
  else writeFile(saveTo, response.body, (error) => {
    if (error) callback(error);
    else callback(null, response.body);
  });
});
Good
function downloadPage(url: string, saveTo: string): Promise<string> {
  return get(url).then(response => write(saveTo, response));
}
downloadPage(url, 'article.html')
  .then(content => console.log(content))
  .catch(error => console.error(error));
HelperDescription
Promise.resolve(value)Wrap a value as a resolved promise
Promise.reject(error)Wrap an error as a rejected promise
Promise.all(promises)Resolves with all results, or rejects on the first rejection, great for running tasks in parallel
Promise.race(promises)Settles as soon as the first promise settles, handy for implementing timeouts

Async/Await are even cleaner than Promises

async/await lets you pause execution at an awaited promise, producing code that reads almost like synchronous code, cleaner than chained .then() calls, and errors can be handled with an ordinary try/catch.

📚 Error Handling thrown errors are a good thing

A thrown error means the runtime successfully caught a problem, stopped execution, and gave you a stack trace to debug from. That's a feature, not a nuisance.

Always use Error for throwing or rejecting

JS/TS technically let you throw or reject with any value, but a plain string caught by higher-level code is confusing to debug. Use Error objects: they carry a .stack property and work naturally with try/catch/finally.

Bad
function calculateTotal(items: Item[]): number {
  throw 'Not implemented.';
}
Good
function calculateTotal(items: Item[]): number {
  throw new Error('Not implemented.');
}

Alternative: skip throw entirely and return typed Result/Failure objects. TypeScript's union types make this ergonomic (e.g. Failable<R, E> = Result<R> | Failure<E>).

Don't ignore caught errors

An empty catch block (or one that just console.logs and moves on) means an error can silently vanish. If you wrap code in try/catch, you're saying "this might fail", so actually handle it (proper logging, a fallback path, etc).

Don't ignore rejected promises

Same principle as above, applied to .catch(): log properly (or handle) instead of a bare console.log, whether using .catch() chains or try/catch around await.

📚 Formatting consistency beats personal preference

Formatting is subjective: don't argue about it, automate it. Use a linter (ESLint for TypeScript, with a shared config like Airbnb's or a Prettier-integrated setup) so the team just agrees to disagree and moves on.

Use consistent capitalization

Casing conventions are subjective, but pick one and apply it everywhere: PascalCase for classes/interfaces/types/namespaces, camelCase for variables/functions/class members, SNAKE_CASE (capitalized) for constants.

Function callers and callees should be close

We read code top-to-bottom like a newspaper, so keep a function's caller directly above (or very near) its callee rather than scattering related methods throughout the class.

Organize imports

Clean import blocks make dependencies scannable at a glance. Good practice:

  • Alphabetize and group imports; remove unused ones
  • Alphabetize named imports (import {A, B, C} from 'foo') and alphabetize sources within each group
  • Prefer import type for type-only imports, avoids dependency cycles since these are erased at runtime
  • Separate groups with blank lines, in this order: polyfills → Node builtins → external modules → internal modules → parent-directory modules → same/sibling-directory modules

Use TypeScript aliases

Configure baseUrl/paths in tsconfig.json to avoid long relative-path imports like ../../../services/UserService.

Bad
import { UserService } from '../../../services/UserService';
Good
import { UserService } from '@services/UserService';
📚 Comments the code should speak for itself

"Don't comment bad code, rewrite it." (Brian W. Kernighan & P. J. Plaugher)

A comment is often an admission that the code itself failed to explain what it does. Code should be the primary source of truth.

Prefer self-explanatory code instead of comments

Bad
// Check if subscription is active.
if (subscription.endDate > Date.now) { }
Good
const isSubscriptionActive = subscription.endDate > Date.now;
if (isSubscriptionActive) { /* ... */ }

Don't leave commented-out code in your codebase

Version control exists precisely so old code doesn't need to linger in comments. Delete it, git history has your back.

Don't have journal comments

Change-log-style comments (// 2016-12-20: Removed monads...) duplicate what git log already gives you for free. Remove them.

Avoid positional markers

Big banner comments (////// Client class //////) mostly add visual noise. Let clear naming, indentation, and your IDE's code-folding do that work instead.

TODO comments

When you need to flag a later improvement, a // TODO: comment is fine, most IDEs surface these in a dedicated list. Just remember: a TODO is not a license to ship bad code.
