SOLID Principles

Five principles of object-oriented class design.

SSingle Responsibility Principlea class should have only one reason to change

A class should do one thing, and only one potential change in the software's specification should be able to affect it. When a class mixes concerns (say, business logic alongside printing or persistence logic), different teams end up editing the same file for unrelated reasons, which makes merge conflicts more frequent and version history harder to read.

Notes
  • Each class should have one, and only one, reason to change.
  • If a class has many reasons to change, it breaks Single Responsibility.
Example: an Invoice class that calculates totals shouldn't also know how to print itself or save itself to a file. Split those into an InvoicePrinter and an InvoicePersistence class instead.
// ❌ Violates SRP: the class handles both report generation and saving
class Report {
  generate(): string { return "Report Content"; }
  saveToFile(content: string): void { console.log(`Saving report: ${content}`); }
}

// ✅ SRP applied: separate concerns into different classes
class ReportGenerator {
  generate(): string { return "Report Content"; }
}
class ReportSaver {
  save(content: string): void { console.log(`Saving report: ${content}`); }
}
OOpen/Closed Principleopen for extension, closed for modification

Classes should be extendable without editing their existing, already-tested code. In practice this means: if you need new behavior, you add a new implementation rather than changing what's already there, usually by depending on an interface or abstract class rather than a concrete one.

Notes
  • In practice: if you need to add new behavior, you shouldn't have to change existing code.
  • Achieved via Interface + Strategy pattern.
  • The Strategy pattern is responsible for choosing the correct instance/implementation to use (which itself respects Single Responsibility).
Example: changing the GOLD tier's pricing logic shouldn't affect the "add to cart" logic. Extension example: adding a new PLATINUM tier means adding a new implementation, not touching the existing tier code.
// ❌ Violates OCP: modifying the class every time a new discount type is added
class Invoice {
  calculateTotal(amount: number, discountType: string): number {
    if (discountType === "percentage") return amount * 0.9;
    if (discountType === "fixed") return amount - 10;
    return amount;
  }
}

// ✅ OCP applied: extend behavior via an interface, no need to modify existing code
interface Discount { apply(amount: number): number; }

class PercentageDiscount implements Discount {
  apply(amount: number) { return amount * 0.9; }
}
class InvoiceProcessor {
  constructor(private discount: Discount) {}
  calculateTotal(amount: number) { return this.discount.apply(amount); }
}
LLiskov Substitution Principlesubtypes must be substitutable for their base type

Named after Barbara Liskov, a mathematician and computer scientist. If you substitute a type with one of its subtypes, the calling code shouldn't need to change or behave unexpectedly. A child class should extend behavior, never narrow it down. The classic trap: a Square that extends Rectangle and overrides setWidth/setHeight to keep both sides equal ends up breaking any code that assumed a Rectangle's width and height are independent.

Notes
  • The calling code should not need to know about the concrete subtype.
  • How to fix violations: push the type-specific logic into the class itself, e.g. a getPrice() method that each subtype implements/overrides.
Example: instead of the caller branching on which concrete subtype it has, give the base type a method like getPrice() and let each subtype override it. The caller just calls getPrice() and never needs to know which subtype it's holding.
// ❌ Violates LSP: Penguin breaks what callers expect from Bird
class Bird { fly(): void { console.log("Flying..."); } }
class Penguin extends Bird {
  fly(): void { throw new Error("Penguins cannot fly!"); }
}

// ✅ LSP applied: separate behaviors properly
abstract class BirdBase { abstract move(): void; }
class FlyingBird extends BirdBase {
  move(): void { console.log("Flying..."); }
}
class NonFlyingBird extends BirdBase {
  move(): void { console.log("Walking..."); }
}
IInterface Segregation Principlemany specific interfaces beat one general-purpose one

Clients shouldn't be forced to implement methods they don't need. A single do-everything interface means that every implementer, even ones that only care about part of it, has to satisfy the whole contract, often with stub methods that throw or do nothing.

Notes
  • About reducing dependencies between components via clear interface contracts.
  • Problem: if an interface changes, its clients are forced to change too.
  • Solution: segregate (split) interfaces to protect clients, depending only on what you actually use.
  • Rule of thumb: depending on something means you're exposed ("subject to") its changes.
Example context: repository interfaces. Split a broad Repository contract into smaller ones (e.g. read-only vs. write) so a client that only reads never has to depend on, or implement, the write methods.
// ❌ Violates ISP: Robot is forced to implement eat() when it doesn't need to
interface Worker { work(): void; eat(): void; }
class RobotWorker implements Worker {
  work(): void { console.log("Working..."); }
  eat(): void { throw new Error("Robots don't eat!"); }
}

// ✅ ISP applied: separate interfaces
interface Workable { work(): void; }
interface Eatable { eat(): void; }

class Human implements Workable, Eatable {
  work(): void { console.log("Working..."); }
  eat(): void { console.log("Eating..."); }
}
class Robot implements Workable {
  work(): void { console.log("Working..."); }
}
DDependency Inversion Principledepend on abstractions, not concrete implementations

High-level modules shouldn't depend on low-level modules. Both should depend on abstractions, and those abstractions shouldn't depend on details. This is usually implemented through Dependency Injection: a high-level class receives the interface it needs from the outside, rather than instantiating a concrete class itself.

Notes
  • Directly linked to Hexagonal Architecture.
  • Distinguishes data flow from dependency flow (they don't have to point the same direction).
  • High-level components should not depend on low-level components.
  • High-level components should be the most stable ones.
  • Business logic should live in the most stable component: the domain.
  • Dependency Injection is one of the main mechanisms used to implement Dependency Inversion.
Example: the domain shouldn't depend on the database driver. Instead, the database adapter depends on a port defined by the domain. See the Hexagonal Architecture & DDD guide for how this plays out across layers.
// ❌ Violates DIP: UserService is tightly coupled to a specific implementation
class MySQLDatabase {
  save(data: string): void { console.log(`Saving "${data}" to MySQL`); }
}
class UserService {
  private db: MySQLDatabase = new MySQLDatabase();
  saveUser(name: string): void { this.db.save(name); }
}

// ✅ DIP applied: depend on an abstraction instead of a concrete class
interface Database { save(data: string): void; }

class MongoDB implements Database {
  save(data: string): void { console.log(`Saving "${data}" to MongoDB`); }
}
class UserServiceDIP {
  constructor(private db: Database) {}
  saveUser(name: string): void { this.db.save(name); }
}
// Usage: new UserServiceDIP(new MongoDB()) can swap databases with zero changes above
