Docker Cheat Sheet

Basics
Basics
docker --version

Shows the installed Docker version.

docker info

Displays detailed system-wide information, including running containers, images, and storage drivers.

docker help

Shows the list of available Docker commands with brief descriptions. You can also use docker <command> --help for help on a specific command.

docker <command> [OPTIONS]

// Example
docker exec -it http-server bash

Basic Docker CLI structure.

Docker's core components:

  • Images: read-only templates used to create containers.
  • Containers: running instances of images, isolated processes.
  • Volumes: persistent storage shared between a container and the host.
  • Networks: allow communication between containers.
  • Dockerfile: a script to build Docker images.
  • Docker Compose: a tool for defining and running multi-container Docker apps.
Images
Images
docker images

Shows a list of images on your local machine, including repository, tag, image ID, and size.

docker pull <image>:<tag>

Pulls an image from Docker Hub. If no tag is provided, Docker uses latest by default.

docker build -t <name>:<tag> <path>

// Example
docker build -t myapp:1.0 .

Builds an image from a Dockerfile. The . refers to the current directory, where the Dockerfile lives.

docker tag <image_id> <repo/name>:<tag>

Adds a tag to an image. Useful for versioning, or preparing to push it to a registry.

docker rmi <image_id or name>

Removes an image. Adding -f forces the removal.

docker inspect <image>

Inspects image details, returning JSON output with layers, config, environment variables, etc.

docker history <image>

Shows the commands used to build the image's layers.

docker push <repo/image>:<tag>

Pushes an image to a registry. You need to be logged in first with docker login.

docker run <image>

Adds a container layer on top of the image and starts it.

Containers
Containers
docker run [OPTIONS] <image>

Creates and starts a new container from the specified image. Common options:

  • -d: run in background (detached)
  • -it: interactive + terminal (great for shells)
  • --rm: automatically remove the container when it stops
  • --name: give the container a name
  • -p <host>:<container>: port mapping
  • -v <host>:<container>: mount a volume
// Only running containers
docker ps

// All (including stopped)
docker ps -a

Lists containers.

docker start <container>

Starts an existing (stopped) container that was created earlier, e.g. via docker run.

docker stop <container>

Stops a container.

docker restart <container>

Restarts a container.

docker rm <container>

Deletes a container.

docker exec -it <container> <command>

// Example
docker exec -it myapp bash

Runs a command inside a running container.

docker attach <container>

Attaches your terminal to a running container's main process, usually the one started with CMD or ENTRYPOINT in the Dockerfile. You'll see live output and can interact if it's an interactive process (like a shell or a foreground server). Detach with Ctrl + P + Q.

docker logs <container>

// In real time
docker logs -f <container>

Shows container logs.

docker inspect <container>

Shows detailed info in JSON format about a container, like network, mounts, config, etc.

docker stats

Shows real-time usage (CPU, memory, I/O) for running containers.

docker container prune

Removes all stopped containers.

Dockerfiles
Dockerfiles

A Dockerfile is a script that automates the creation of a Docker image by specifying the base image, what files to copy, what commands to run, and the environment configuration.

Common instructions:

  • FROM: the base image to use (must be the first line)
  • RUN: executes commands in a new image layer
  • CMD: the default command to run when the container starts
  • COPY: copies files/directories from the host into the image
  • ADD: like COPY, but can also extract tar files or download from URLs
  • WORKDIR: sets the working directory inside the container
  • ENV: sets environment variables
  • EXPOSE: documents which port the container will listen on
  • ENTRYPOINT: configures the container to run as an executable
# Use base image
FROM node:18

# Set working directory
WORKDIR /app

# Copy files
COPY package*.json ./
RUN npm install

COPY . .

# Expose port
EXPOSE 3000

# Default command
CMD ["npm", "start"]

An example Dockerfile: this builds a Node.js app image starting from a clean Node.js 18 environment, installs dependencies, adds the app's source code, and runs the app on port 3000 using npm start.

docker build -t myapp:1.0 .

Builds an image.

# Stage 1 - Builder
FROM node:18 AS builder
WORKDIR /app
COPY package*.json ./
RUN npm install
COPY . .
RUN npm run build

# Stage 2 - Minimal runtime
FROM nginx:alpine
COPY --from=builder /app/build /usr/share/nginx/html

An example of a multi-stage build: it uses multiple FROM statements in one Dockerfile, each starting a new stage, letting you copy only what's needed from earlier stages into the final image. This keeps the final image small (no dev tools, compilers, or source code), reduces the attack surface, and makes builds cleaner and more modular. In this example, the final image only contains the compiled application (e.g. Vue or React), with no Node.js or source code left in it.

Volumes
Volumes
docker volume create <volume_name>

Creates a volume.

docker volume ls

Lists volumes.

docker volume rm <volume_name>

Removes a volume. Only works if the volume isn't being used by any container.

docker volume prune

Deletes all unused volumes.

docker run -v <volume_name>:<container_path> <image>

Uses a volume in a container.

docker volume inspect <volume_name>

Shows details of a volume, such as its path on the host, mountpoint, etc.

docker run -v /home/user/app:/usr/src/app node

Example of a bind mount: great for local development, since changes in the code editor reflect instantly inside the container. It's not managed by Docker.

docker volume create pgdata
docker run -v pgdata:/var/lib/postgresql/data postgres

Example of a named volume: data remains even if the container is deleted, and it's easy to back up and manage via the Docker CLI since it's managed by Docker. Useful whenever you want Docker-managed, persistent, portable data.

services:
  frontend:
    image: node:lts
    volumes:
      - myapp:/home/node/app
volumes:
  myapp:

Example of how to use a volume with Docker Compose.

Networks
Networks

Common network types:

  • bridge: the default for standalone containers, an isolated, local container network.
  • host: shares the host's network stack, no network isolation.
  • none: completely isolates the container's networking stack; only the loopback device exists inside it.
  • overlay: used for multi-host (swarm) networks, spanning across multiple Docker daemons.
  • macvlan: assigns a MAC address to a container, so it looks like a physical device on the LAN.
docker network ls

Lists networks.

docker network inspect <network_name>

Gives detailed information about a network: driver, scope, IP address details, the list of containers connected to it, etc.

docker network create <network_name>

// Example
docker network create \
  --driver bridge \
  --subnet 192.168.100.0/24 \
  --gateway 192.168.100.1 \
  my_custom_net

Creates a network.

docker network rm <network_name>

Removes a network.

docker network create mynet
docker run -d --name db --network mynet postgres
docker run -d --name app --network mynet myapp

Example of using a custom network: attaching containers to the same custom bridge network allows automatic name-based DNS resolution between them. Here, app can reach db just by using db:5432.

services:
  app:
    image: myapp
    networks:
      - frontend
  db:
    image: postgres
    networks:
      - backend
networks:
  frontend:
  backend:

Docker Compose automatically creates a custom network for the services, but you can still specify your own network if needed.

Docker Compose
Docker compose
services:
  web:
    build: ./web
    ports:
      - "3000:3000"
    depends_on:
      - db
  db:
    image: postgres:14
    environment:
      POSTGRES_PASSWORD: secret
    volumes:
      - dbdata:/var/lib/postgresql/data

volumes:
  dbdata:

Example of a docker-compose.yml file: the services part contains each containerized app you run, volumes contains the shared persistent storage, and networks contains the custom container communication layer.

Most common commands:

  • docker compose up: start all services
  • docker compose up -d: start in detached mode
  • docker compose down: stop and remove containers, network, volumes
  • docker compose ps: list running services
  • docker compose logs: show logs for all services
  • docker compose build: build services from Dockerfiles
  • docker compose restart: restart one or more services
  • docker compose top: display processes running inside containers
  • docker compose run --rm: run a temporary container and remove it after it exits
docker compose -f docker-compose.yaml --env-file env.env -p stackname up -d

Starts all services defined in docker-compose.yaml, using environment variables from env.env and stackname as the project name (which prefixes container, volume, and network names), running in the background since -d is set. A way to launch a full multi-container application with custom naming.

// .env file
POSTGRES_PASSWORD=secret

// Reference in the Docker Compose file
environment:
  - POSTGRES_PASSWORD=${POSTGRES_PASSWORD}

It's good practice to use .env files for secrets and configuration.

docker compose down --volumes --rmi all --remove-orphans

Shuts down the Compose application and removes everything related to it: containers, networks, volumes, images, and even stray orphaned containers. Essentially a full cleanup/reset of the stack.

docker compose up --build --force-recreate

Rebuilds the service images and recreates all containers from scratch, regardless of whether anything changed. A clean way to make sure you're running the latest build and configuration for every service.

docker-compose down -v

Stops the Compose app and cleans up its containers, networks, and associated volumes, without removing images or affecting other projects. Useful for resetting the app and starting fresh without leftover data.

Cleanup
Cleanup
docker stop $(docker ps -q)

Stops all running containers.

docker rm $(docker ps -a -q)

Removes all containers.

docker image prune

Removes all unused images.

docker rmi $(docker images -q)

Removes all images.

docker volume prune

Removes unused volumes.

docker volume rm $(docker volume ls -q)

Removes all volumes.

docker network prune

Removes unused networks.

docker system prune

Removes everything not in use. Add -a to also remove unused images, not just dangling ones (images not tagged and not referenced by any container). Add --volumes to clean up volumes as well.

docker system df

Shows the space used by containers, images, volumes, and build cache, and what's reclaimable.
