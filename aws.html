AWS Cheat Sheet

Security & Compute
Security

IAM

Think of IAM as the bouncer standing in front of every other AWS service. Nothing happens, not an API call, not a console click, until IAM checks whether you're allowed to do it. It doesn't cost anything and covers your whole account.

  • Users are individual logins, usually people, sometimes with their own access keys.
  • Groups let you hand the same permissions to a bunch of users at once, like "everyone on the backend team".
  • Roles are permissions you borrow temporarily instead of owning. EC2 and Lambda lean on these constantly.
  • Policies are just JSON documents saying what's allowed or denied on which resources.
  • Turn on MFA, especially for the root account. And stick to least privilege: give people only what they actually need.
Example: a policy that only allows s3:PutObject and s3:GetObject on one bucket, handed to the developers group.
In practice: instead of pasting AWS keys into your code (please don't), give the EC2 instance a role. AWS quietly rotates short lived credentials into it, and your app never touches a long term secret.
Reach for it: any time a person, app, or service needs some level of access to your AWS account. So, basically always.
Compute

EC2

EC2 is a virtual machine you can SSH into, install whatever you want on, and treat pretty much like a normal server. AWS worries about the physical hardware underneath, you handle everything above the OS.

  • The instance type decides your CPU/RAM/network mix, things like t3.medium for general use or c7g if you need raw compute.
  • An AMI is the template your instance boots from, an OS plus whatever software is baked in.
  • Security Groups act as a stateful firewall right on the instance, you list which ports and sources get through.
  • Pricing comes in a few flavors: On-Demand (pay as you go), Reserved or Savings Plans (cheaper if you commit), and Spot (way cheaper, but AWS can take it back with short notice).
Example: launch a t3.medium Ubuntu box, open port 443 in its Security Group, and attach an Elastic IP so the address survives a reboot.
In practice: an animation studio renders frames on Spot instances since losing one mid job just means retrying elsewhere, but keeps its public facing web servers on On-Demand for guaranteed uptime.
Reach for it: whenever you need a general purpose server and want full control over the OS.
Storage
Block storage

EBS

EBS is a virtual hard disk that plugs into a single EC2 instance, same idea as a physical drive plugging into a physical computer. It's block storage, so the OS just sees a raw disk it can format however it likes. Unlike S3, which only knows how to store whole objects.

  • Lives in one Availability Zone and normally attaches to just one instance at a time.
  • Outlives the instance, stop or terminate it and the volume sticks around, ready to move elsewhere in the same AZ.
  • Snapshots back up the volume's state to S3 behind the scenes, and can spin up new volumes or copy across regions.
  • Volume types trade speed for cost: gp3 is the default general purpose SSD, io2 is for demanding databases, st1/sc1 are cheaper spinning disk options.
Example: a database instance keeps its root disk on gp3 but puts the actual data files on a separate io2 volume for better performance.
In practice: before a risky deploy, an ops team snapshots the volume first. If things break, they spin up a fresh volume from that snapshot and roll back in minutes.
Reach for it: anytime an EC2 instance needs fast, persistent disk, databases, boot volumes, that kind of thing.
File storage

EFS

EFS is a shared network drive that many EC2 instances, even ones in different Availability Zones, can mount at once. Where EBS is one disk tied to one server, EFS is one shared drive used by many servers, and it grows or shrinks on its own with no capacity to plan ahead of time.

  • Hundreds of instances (or Lambda functions) can mount it simultaneously and see the same files.
  • Fully elastic, you pay for what you store, no pre-allocating size like with EBS.
  • Spans multiple AZs in a region, unlike EBS which sticks to one.
  • A bit slower per operation than EBS, but it's built for shared access, not raw single instance speed.
Example: ten web servers behind a load balancer all mount the same EFS filesystem, so a file uploaded through one server shows up instantly on the rest.
In practice: a CMS running on Auto Scaling needs every new instance to have the same media library the moment it launches, EFS handles that without any syncing.
Reach for it: whenever multiple instances or functions need to read and write the same files at once.
Object storage

S3

S3 stores files in buckets. It isn't a real filesystem, there's no true folder structure, just object keys that can contain slashes so they look like paths. Everything happens over HTTP, so any number of clients can read and write at once without it being tied to one machine.

  • Rated for eleven nines of durability, your data gets copied across multiple facilities automatically.
  • Storage classes trade cost for retrieval speed: Standard for frequent access, Standard-IA for infrequent, Glacier for cheap long term archiving.
  • Lifecycle rules move or delete objects automatically as they age, no manual cleanup needed.
  • Can serve static websites directly and pairs naturally with CloudFront.
Example: a bucket called my-app-uploads holding an object at users/1234/avatar.png. No real folder, just a key with a slash in it.
In practice: a media company leaves fresh video in Standard while editors work on it, then a lifecycle rule quietly ships anything untouched for 90 days off to Glacier.
Reach for it: durable, essentially unlimited storage for backups, media, logs, static assets, or data lakes.
Networking
Networking

VPC

A VPC is your own isolated slice of AWS's network, basically your private data center, only software defined. You decide the IP ranges, how it's split into subnets, and exactly what can talk to what, inside the VPC and out to the internet.

  • Subnets split the VPC by Availability Zone. A public subnet routes to an Internet Gateway, a private one doesn't.
  • Internet Gateway gives resources in a public subnet two way internet access.
  • NAT Gateway sits in a public subnet so private subnet resources can reach out (say, for updates) without being reachable from outside.
  • Route tables and Network ACLs control traffic at the subnet level, on top of instance level Security Groups.
Example: a load balancer sitting in a public subnet, with the app servers behind it tucked into a private subnet only reachable from the load balancer.
In practice: a fintech puts its RDS database in a private subnet with no route to the internet at all. Even a compromised app server can't reach it directly from outside the VPC.
Reach for it: pretty much any AWS deployment sits inside one, it's the default networking foundation.
Content delivery

CloudFront

CloudFront is AWS's CDN, a global set of edge locations that cache your content physically close to your users, so requests don't have to travel all the way back to your origin every single time.

  • The first request from a region hits the origin (S3, EC2, any HTTP server) and gets cached at the nearest edge. Everyone after that gets served from cache.
  • Cuts latency for users and load on your origin at the same time.
  • Throws in free HTTPS certs, DDoS protection via Shield, and geo restriction if you need it.
Example: a product photo stored in an S3 bucket in Virginia gets cached at a Tokyo edge location the first time someone there requests it, everyone after that gets it almost instantly.
In practice: a streaming platform puts CloudFront in front of its video origin. Because most requests get served from edge cache, load times and origin costs both drop noticeably.
Reach for it: users spread across the globe and you want faster loads with less strain on your origin.
DNS

Route 53

Route 53 is AWS's DNS service, it turns domain names into the IP addresses computers actually use, and can register new domains too. Beyond plain DNS, it can route smartly based on where a user is, how healthy each endpoint looks, and whatever strategy you pick.

  • Simple routing: one record, one destination (or a fixed set).
  • Weighted routing: split traffic by percentage, handy for gradual rollouts.
  • Latency based routing: send each user to whichever region answers fastest for them.
  • Failover routing: shift traffic to a backup automatically if health checks catch a problem.
  • Geolocation routing: serve different content depending on where the request comes from.
Example: a weighted policy sending 90% of traffic to production and 10% to a canary build being tested.
In practice: a SaaS company runs servers in the US and Europe. Latency based routing sends each visitor to whichever is faster, and if the US region fails its health check, failover routing shifts everyone to Europe until it's back.
Reach for it: managing domain names or routing traffic intelligently across regions.
Databases
Managed database

RDS

RDS runs standard database engines (MySQL, PostgreSQL, MariaDB, SQL Server, Oracle) so you don't have to. Backups, patching, monitoring, failover, all handled. You still pick the engine and instance size, but you never SSH in to babysit the server itself.

  • Multi-AZ: AWS keeps a synchronous standby in a different AZ purely for availability. Primary fails, RDS fails over automatically, usually within a minute or two.
  • Read Replicas: async, read only copies for scaling reads. They don't fail over automatically by default, a classic gotcha.
  • Automated backups and point in time restore come built in.
Example: a Multi-AZ Postgres instance fails over to its standby in another AZ the moment the primary hits a hardware issue, no app changes required.
In practice: an online store adds two Read Replicas and points all the catalog browsing traffic at them, keeping the primary free to handle checkouts.
Reach for it: you want a standard SQL database and don't want to manage the server underneath.
Managed database

Aurora

Aurora is AWS's own database engine, wire compatible with MySQL and Postgres but rebuilt underneath for the cloud. Compute and storage are separated, the storage layer replicates data six ways across three Availability Zones on its own, which is what lets Aurora recover faster and scale further than plain RDS.

  • Storage scales up to 128TB automatically, nothing to provision by hand.
  • Supports up to 15 read replicas with much less lag than standard RDS, and any of them can be promoted to primary during a failover.
  • Aurora Serverless scales compute up and down, even to zero, based on load. Good for spiky or unpredictable workloads.
  • Generally faster and pricier than equivalent RDS, partly billed on storage I/O.
Example: an Aurora Postgres cluster with one writer and three readers sharing the same storage volume, a reader can become writer in seconds during a failover.
In practice: a SaaS product that's busy during the day and quiet overnight runs Aurora Serverless v2 so capacity scales down automatically instead of paying for a fixed size instance around the clock.
Reach for it: you want MySQL or Postgres compatibility with better scaling and faster failover than RDS, and the cost bump is fine.
Traffic & Serverless
Traffic & scaling

ELB & Auto Scaling

These two nearly always show up together. The load balancer sits in front of your servers, spreads incoming requests across them, and keeps health checking so it can route around anything that's failing. The Auto Scaling Group watches metrics like CPU and launches or kills EC2 instances to match demand, and the load balancer automatically starts sending traffic to whatever it adds.

  • ALB (Application Load Balancer): layer 7, routes by URL path or host, the usual pick for web apps.
  • NLB (Network Load Balancer): layer 4, extremely low latency, for non-HTTP or performance heavy traffic.
  • An ASG defines min, desired, and max instance counts along with scaling policies, like a target CPU percentage.
  • Unhealthy instances get replaced automatically, so the whole thing self heals.
Example: an ASG set to min 2 / max 10 adds two more instances whenever average CPU crosses 70%, and the ALB starts routing to them as soon as they pass health checks.
In practice: a news site gets hit by a viral story. The ASG notices CPU climbing and launches new instances within minutes, then scales back down once things settle to keep costs sane.
Reach for it: your app needs to survive a server dying and handle traffic that swings around unpredictably.
Serverless

Lambda

Lambda runs your code in response to events, and you never touch a server. Upload a function, and AWS handles allocating compute, scaling it up for lots of simultaneous invocations, and scaling back to zero the moment there's nothing to do. You're billed by the millisecond of actual execution, not for sitting idle.

  • Triggered by all sorts of events: HTTP requests through API Gateway, S3 uploads, scheduled EventBridge rules, SQS/SNS messages, and more.
  • Capped at 15 minutes per invocation, not built for long running or persistent processes.
  • Scales near instantly for concurrent invocations, within your account's concurrency limits.
  • Cold starts, that extra bit of latency on the first call after idling, are the known trade off, especially for heavier runtimes.
Example: a photo upload to S3 triggers a Lambda that resizes it into a thumbnail and saves it to another bucket, no server ever sits around waiting.
In practice: a company builds its entire API out of API Gateway plus Lambda, one function per endpoint. Invocation volume spikes 50x during a big sale and Lambda just scales, no infrastructure planning needed, and cost drops back to near nothing when things are quiet.
Reach for it: short, event driven tasks where you'd rather not manage servers at all.
