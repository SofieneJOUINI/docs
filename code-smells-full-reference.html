Code Smells & Heuristics

Robert C. Martin's full catalog from Clean Code: every smell and heuristic, organized by category and color-coded for fast scanning.

🔴 Core Hygiene comments, build/test environment, function basics

Comments

C1

Inappropriate Information

Don't use comments to hold information that belongs in a different system, like version control or an issue tracker. Change logs, author names, and last-modified dates just clutter the file; comments should hold technical notes about the code and design, nothing else.

C2

Obsolete Comment

A comment that's gone stale, irrelevant, or wrong is worse than no comment: it actively misleads. Update or delete a comment the moment you notice it's drifted from the code it describes.

C3

Redundant Comment

A comment is redundant if it says no more than the code already says on its own (e.g. i++; // increment, or a Javadoc that just repeats the method signature). Comments should say something the code itself cannot.

C4

Poorly Written Comment

If a comment is worth writing, it's worth writing well: correct grammar, careful word choice, brevity, no stating the obvious.

C5

Commented-Out Code

Nobody deletes commented-out code because everyone assumes someone else needs it, so it just rots: calling functions that no longer exist, following conventions that are long gone. Delete it; version control remembers it even if the file doesn't.

Environment

E1

Build Requires More Than One Step

Building a project should be one trivial operation: a single command to check out the system, another single command to build it. No hunting for stray JARs or arcane multi-step scripts.

E2

Tests Require More Than One Step

You should be able to run every unit test with one command, ideally one click in the IDE. Running the full suite is so fundamental that it needs to be quick, easy, and obvious.

Functions

F1

Too Many Arguments

Fewer arguments is better: zero is ideal, then one, then two, then three. More than three is highly questionable and should be avoided.

F2

Output Arguments

Readers expect arguments to be inputs, not outputs. If a function must change the state of something, have it change the state of the object it's called on instead of writing back into a passed-in argument.

F3

Flag Arguments

A boolean argument is a loud admission that the function does more than one thing, confusing at the call site and a sign it should be split into two functions.

F4

Dead Function

Methods that are never called should be deleted outright. Don't be sentimental about it, source control still remembers them.

🟢 General the largest category, day-to-day design heuristics (G1-G36)

G1 – G12

G1

Multiple Languages in One Source File

Modern files can mix languages freely (Java + XML + HTML + JS in one JSP, for example), which is confusing at best. Aim for one language per source file, and minimize the extent of any unavoidable extras.

G2

Obvious Behavior Is Unimplemented

Follow the "Principle of Least Surprise": implement the behavior another programmer would reasonably expect from a function's name (e.g. a day-name parser should handle abbreviations and ignore case). Skipping obvious behavior erodes trust in the code and forces readers back into the implementation details.

G3

Incorrect Behavior at the Boundaries

Correct behavior is more complicated than it looks. Don't trust intuition over verification. Hunt down every boundary condition and corner case, and write a test for each one.

G4

Overridden Safeties

Turning off compiler warnings, disabling failing tests "to fix later," or manually overriding safety mechanisms is always risky: it trades a short-term inconvenience for a much larger, later cost.

G5

Duplication

One of the most important rules here: "Once, and only once" (Kent Beck's phrasing). Every duplication is a missed abstraction opportunity: identical clumps of code become a shared method, repeated switch/if chains become polymorphism, and similar-but-not-identical algorithms become a Template Method or Strategy pattern.

G6

Code at Wrong Level of Abstraction

Higher-level and lower-level concepts need to be cleanly separated, usually into a base class and its derivatives. If a base class carries a detail that not every derivative can support (e.g. a percentFull() method only some stacks can meaningfully implement), that's a sign the abstraction boundary is misplaced. Faking an answer (like returning 0) just tells a lie instead of fixing the abstraction.

G7

Base Classes Depending on Their Derivatives

Base classes should generally know nothing about their derivatives, so the two can be deployed and modified independently. Exceptions exist (e.g. tightly-fixed finite state machines), but as a default, keeping bases ignorant of derivatives lowers the blast radius of any change.

G8

Too Much Information

Well-defined modules expose small interfaces and hide everything else: data, utility functions, constants, temporaries. Fewer public methods and instance variables means lower coupling and an easier-to-use module.

G9

Dead Code

Code that's never executed (an impossible if branch, a catch for an exception that's never thrown) rots quietly: it still compiles but stops tracking newer conventions. Delete it the moment you find it.

G10

Vertical Separation

Variables and functions should be defined close to where they're used: local variables just above their first use, private functions just below their first call, so a reader can find things by scanning nearby, not hunting hundreds of lines away.

G11

Inconsistency

If you do something a certain way once, do all similar things the same way: consistent naming and structure (the same variable name for the same kind of object, a matching naming pattern across similar methods) makes code far easier to read and modify.

G12

Clutter

Empty default constructors, unused variables, dead functions, comments that add nothing, all clutter, all removable. Keep source files clean and free of anything that doesn't earn its place.

G13 – G24

G13

Artificial Coupling

Don't couple two things that don't actually depend on each other, e.g. a general-purpose enum or static function shoved inside a more specific class "for convenience." Take the time to figure out where something really belongs instead of dropping it wherever's easiest right now.

G14

Feature Envy

A method that reaches into another object's accessors/mutators to do its work "envies" that other class's scope: it really wants to live inside it. Usually worth eliminating, though sometimes it's a necessary trade-off (e.g. keeping report-formatting logic out of a domain class, even at the cost of some envy).

G15

Selector Arguments

A trailing boolean (or enum/int) argument that "selects" which behavior to run is a lazy substitute for splitting the function into several smaller, clearly-named functions. Prefer many small functions over one function with a hidden mode switch.

G16

Obscured Intent

Run-on expressions, Hungarian notation, and magic numbers all bury the author's intent. It's worth the extra time to make code's purpose visible on the surface, not just recoverable by close reading.

G17

Misplaced Responsibility

Code should live where a reader would naturally expect to find it (the "Principle of Least Surprise" again): a constant like PI belongs with the trig functions, not scattered wherever was convenient. When performance forces a less-obvious placement, the function's name should say so explicitly.

G18

Inappropriate State (Static vs. Non-Static)

A function is a reasonable candidate for static only if there's no chance you'll ever want it to behave polymorphically. When in doubt, keep it non-static: it's easy to convert a non-static method to static later, much harder to go the other way once callers depend on it.

G19

Use Explanatory Variables

Break calculations into intermediate values held in well-named variables. It's one of the most effective ways to make an otherwise-opaque calculation instantly readable. More explanatory variables is almost always better than fewer.

G20

Function Names Should Say What They Do

A call like date.add(5) tells you nothing about units or whether the original is mutated. If the function mutates in place, name it like increaseByDays; if it returns a new value, name it like daysLater. If you have to check the implementation to know what a function does, its name has failed.

G21

Understand the Algorithm

Getting a function to pass its tests isn't the same as understanding why it works. Before considering a function done, make sure you actually understand its logic; often the best way to build that understanding is to refactor it until the code itself makes the algorithm obvious.

G22

Make Logical Dependencies Physical

If module A depends on assumptions about module B (a "logical" dependency, like assuming a fixed page size), that assumption should instead be explicitly requested from B (a "physical" dependency, like calling B.getMaxPageSize()), otherwise a silent, fragile coupling exists that breaks the moment B's assumptions change.

G23

Prefer Polymorphism to If/Else or Switch/Case

Most switch statements are the "obvious brute-force" choice, not the right one: consider polymorphism first. Rule of thumb: at most one switch statement per type of selection, and its cases should just construct the polymorphic objects that replace every other switch on that same type elsewhere in the system.

G24

Follow Standard Conventions

A team should agree on a coding standard (naming, brace placement, etc.) based on common norms, and every member should follow it, so the code itself should be example enough, with no separate document required.

G25 – G36

G25

Replace Magic Numbers with Named Constants

Raw numbers in code should almost always hide behind a named constant (SECONDS_PER_DAY instead of 86400). "Magic" applies to any value that isn't self-describing, not just numbers: an unexplained string ID in a test assertion is just as opaque as an unexplained number. A few extremely well-known constants (like Math.PI) are the rare exception worth naming anyway, purely because the risk of a typo (e.g. a transposed digit) is otherwise invisible.

G26

Be Precise

Vague decisions in code (assuming a query returns only one match, using floats for currency, skipping locking around concurrent updates) are either laziness or unresolved disagreement, and both should be eliminated. Know exactly why you made each decision and how edge cases are handled.

G27

Structure over Convention

Enforce a design decision with structure, not just convention: an abstract base class with abstract methods forces every subclass to implement them; a switch/case relying on a naming convention enforces nothing.

G28

Encapsulate Conditionals

Extract a named function for a compound boolean condition (if (shouldBeDeleted(timer))) rather than inlining the raw logic (if (timer.hasExpired() && !timer.isRecurrent())): the intent reads instantly.

G29

Avoid Negative Conditionals

Positive conditionals are easier to parse than negative ones: prefer if (buffer.shouldCompact()) over if (!buffer.shouldNotCompact()).

G30

Functions Should Do One Thing

A function with multiple sequential sections (loop, check, act) is really doing several things. Split each section into its own well-named function so each one does exactly one thing.

G31

Hidden Temporal Couplings

When functions must be called in a specific order, make that order impossible to get wrong: have each step's output become the next step's required input (a "bucket brigade"), instead of relying on silent, undocumented sequencing.

G32

Don't Be Arbitrary

Every structural decision should have a reason, and that reason should be visible in the structure itself. Nesting a public class inside another class "just because it was convenient", when unrelated code needs it too, creates confusion with no benefit; follow the convention instead (top-level, package-scoped).

G33

Encapsulate Boundary Conditions

Boundary math like level + 1 is easy to lose track of if it's scattered across a function. Capture it once in a clearly-named variable (nextLevel) instead of repeating the raw expression.

G34

Functions Should Descend Only One Level of Abstraction

Every statement inside a function should sit one level of abstraction below what the function's name promises. Mixing a high-level concept (e.g. "a rule has a size") with low-level syntax details (raw HTML tag construction) in the same function is a common, easy-to-miss violation, and separating them often surfaces bugs hiding in the mixed-up code.

G35

Keep Configurable Data at High Levels

Defaults and configuration values that are meaningful at a high level shouldn't be buried inside a low-level function. Expose them as arguments passed down from the top, so they stay easy to find and change.

G36

Avoid Transitive Navigation

Also known as the Law of Demeter / "writing shy code": a module should only know about its immediate collaborators, never chain through them to reach a collaborator's collaborator (a.getB().getC().doSomething()). Long navigation chains make the architecture rigid: every intermediate change forces updates everywhere that chain appears.

🟣 Language & Naming Java-specific habits, and naming things well

Java

J1

Avoid Long Import Lists by Using Wildcards

Importing two or more classes from the same package with a wildcard keeps the top of a file concise. Wildcard imports also create weaker (non-hard) dependencies than a long list of specific imports, keeping modules less coupled, at the minor cost of occasional name collisions across packages.

J2

Don't Inherit Constants

Gaining access to constants by inheriting an interface that only exists to hold them is a hideous workaround for scoping rules: it hides where the constants actually come from. Use a static import instead.

J3

Constants versus Enums

Once a language supports enums, prefer them over public static final int constants: an int's meaning can get lost, while an enum is self-naming and can carry its own methods/fields for far more expressive, flexible code.

Names

N1

Choose Descriptive Names

Names carry roughly 90% of what makes software readable. Don't rush past this. Meanings drift as software evolves, so periodically re-evaluate whether existing names still fit.

N2

Choose Names at the Appropriate Level of Abstraction

Don't let a name leak implementation details it shouldn't commit to. A Modem interface built entirely around "phone numbers" breaks the moment a cable or USB modem needs to implement it. A more abstract term like connectionLocator keeps the interface honest about what it actually represents.

N3

Use Standard Nomenclature Where Possible

Reuse existing naming conventions wherever they exist: pattern names (e.g. Decorator) belong in the names of classes implementing that pattern, language idioms (toString) should be followed rather than reinvented, and a project's own ubiquitous language should be used consistently throughout the code.

N4

Unambiguous Names

A name should make a function's purpose unambiguous. A vague name like doRename next to an internal call to renamePage gives the reader nothing to distinguish them. A longer, fully explicit name (renamePageAndOptionallyAllReferences) pays for its length with clarity.

N5

Use Long Names for Long Scopes

Name length should track scope length. A single-letter loop variable is perfectly fine across five lines; the same short name loses all meaning if its scope spans hundreds.

N6

Avoid Encodings

Type or scope prefixes (Hungarian notation like m_, or subsystem tags like vis_) are noise in modern tooling: today's environments already surface that information without mangling the name itself.

N7

Names Should Describe Side-Effects

A name shouldn't undersell what a function actually does. A getter that lazily creates the value if it doesn't exist yet is doing more than "getting": a name like createOrReturnOos is more honest than a bare getOos.

🟠 Tests a thorough, fast, revealing test suite
T1

Insufficient Tests

"That seems like enough" isn't a real metric. A test suite is insufficient as long as there's any condition that hasn't been explored or any calculation that hasn't been validated.

T2

Use a Coverage Tool!

Coverage tools surface exactly which lines/branches are untested. Most IDEs highlight covered vs. uncovered code visually, making gaps (like an untested catch body) trivial to spot.

T3

Don't Skip Trivial Tests

Trivial tests are cheap to write and their documentary value is higher than the cost of writing them, so keep them.

T4

An Ignored Test Is a Question about an Ambiguity

When requirements are genuinely unclear, express that uncertainty as a commented-out or @Ignored test: a concrete question about the ambiguity, not a silent gap.

T5

Test Boundary Conditions

The middle of an algorithm is usually right; boundaries are where mistakes hide, so give them special attention.

T6

Exhaustively Test Near Bugs

Bugs cluster. Once you find one in a function, thoroughly test the rest of that function: it's rarely alone.

T7

Patterns of Failure Are Revealing

A pattern across failing tests (e.g. every test with an input over five characters fails) is often the fastest route to a diagnosis, another reason to keep test cases complete and well-ordered.

T8

Test Coverage Patterns Can Be Revealing

Looking at what code the passing tests do and don't execute gives clues about why the failing tests are failing.

T9

Tests Should Be Fast

A slow test is a test that eventually stops getting run. When time gets tight, slow tests are the first thing dropped from the suite. Keep them fast.

📚 Design Patterns reusable solutions to recurring design problems

Design patterns are proven, reusable solutions to recurring software design problems: they promote readability, maintainability, and flexibility, but should be applied judiciously, with a clear understanding of what each one costs as well as what it buys. Patterns are generally grouped by intent:

Creational: mechanisms for object creation that increase flexibility and reuse.

Structural: ways to assemble objects and classes into larger structures while keeping them flexible and efficient.

Behavioral: patterns concerned with effective communication and division of responsibility between objects.
